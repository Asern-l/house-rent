## 第一部分：必做基础功能（项目主干，必须完整实现）

以下五个模块构成系统的核心骨架，缺一不可。每个模块包含“目的”、“实现逻辑（文字步骤）”和“数据库/合约交互说明”。

### 1. 用户身份注册与登录（简化版）

**目的**：识别用户身份，区分房东与租客，为后续合同签署和支付提供账户基础。因无法对接公安实名接口，本系统采用“手机号+验证码”的简化认证，并允许用户绑定区块链钱包地址。

**实现逻辑（文字步骤）**：

1. 前端提供注册表单：手机号、密码（或短信验证码）、角色（房东/租客）。可选：绑定MetaMask钱包地址（若不绑定，系统可自动生成一个临时地址用于演示）。
2. 后端接收请求后，检查手机号是否已注册。
3. 生成唯一用户ID（例如 `uid_时间戳_随机数`）。
4. 将用户信息存入数据库：用户ID、手机号、密码哈希、角色、注册时间、钱包地址（若填写）。
5. 若未绑定钱包，系统为该用户分配一个测试网地址（或提示用户安装MetaMask）。
6. 返回登录凭证（token），前端保存。

**数据库表结构（逻辑）**：

- 用户表：`user_id, phone, password_hash, role, wallet_address, created_at`

**注意事项**：本系统不模拟身份证上传，答辩时说明“生产环境需接入合法实名认证服务”。

### 2. 房源发布与AI图片可信度检测

**目的**：房东上传房源信息和图片，系统自动检测图片是否造假（PS、盗图），给出可信度分数，并将分数和图片哈希上链存证。

**实现逻辑（文字步骤）**：

1. 房东登录后，进入“发布房源”页面，填写：地址（可调用地图API建议）、租金、户型、描述，上传最多9张图片。
2. 后端校验房东身份（角色为“房东”且已登录）。
3. 对每张图片执行伪造检测：
   - 模拟方式：编写规则函数，依据图片尺寸、清晰度、EXIF信息等计算分数（0-100）。例如：尺寸大于800x600得20分，无异常边缘得30分，颜色分布正常得30分，综合得出分数。
   - 或调用免费开源库（如`imagemagick`检测元数据）。
   - 为演示稳定，可手工预设“合格”图片得80分以上，“不合格”得40分。
4. 计算所有图片的平均分。若平均分低于60，拒绝发布，返回“图片可信度不足，请更换真实照片”。
5. 计算每张图片的SHA-256哈希值，得到哈希列表。
6. 生成房源唯一ID（`lst_时间戳`）。
7. 调用智能合约的 `storeListing` 函数传入：房源ID、平均分、图片哈希列表、房东钱包地址。合约记录存证并触发事件。
8. 等待交易确认，获取交易哈希。
9. 将房源信息存入数据库：房源ID、房东ID、地址、租金、描述、平均分、图片哈希列表、链上交易哈希、状态（已上架）。
10. 返回发布成功及房源ID。

**智能合约功能描述**：

- 合约提供 `storeListing` 方法，参数为 `string listingId, uint256 aiScore, bytes32[] imageHashes`。
- 合约内部存储这些数据，并发出 `ListingStored` 事件。

**前端展示**：房源详情页显示“AI可信度：XX分（可信/可疑）”，并提供链上存证哈希的链接（指向区块浏览器）。

### 3. 电子合同签署与链上存证（核心亮点）

**目的**：租客与房东在线签署租赁合同，将合同全文的哈希值上链存证，确保内容不可篡改、签署行为不可抵赖。本模块遵循《电子签名法》关于“可靠电子签名”的基本要求：双方实名（本系统用手机号代替）、签署意愿确认、合同内容固化。

**实现逻辑（文字步骤）**：

1. 租客在房源详情页点击“申请租房”，系统创建合同草稿。合同内容基于模板自动生成，包含：房源地址、租金、押金、租期起止、双方手机号、双方钱包地址、签约日期等。
2. 租客确认合同内容无误后，点击“同意签署”。系统记录租客的签署时间、IP地址，并将合同状态更新为“租客已签”。
3. 房东登录后，在“待签署合同”列表中看到该合同，点击“同意签署”。系统记录房东签署时间、IP地址。
4. 双方签署完成后，后端将合同全部字段拼接成一个规范字符串（例如JSON格式），计算该字符串的SHA-256哈希值。
5. 调用智能合约的 `storeContract` 函数，传入：合同ID（系统生成）、合同哈希、租客钱包地址、房东钱包地址。合约存储并发出事件。
6. 获得链上交易哈希后，更新数据库合同状态为“已生效”，并保存交易哈希。
7. 可选：生成PDF格式的电子合同文件，包含合同正文、双方签署时间、链上存证二维码，提供下载。

**智能合约功能描述**：

- 合约提供 `storeContract` 方法，参数为 `string contractId, bytes32 contractHash, address tenant, address landlord`。
- 合约内部记录上述信息，并确保同一个合同ID只能存储一次。

**前端交互**：双方需使用MetaMask对合同内容进行签名（或简单点击确认，模拟签名意愿）。为简化，可设计为“后台自动调用合约”，但强调演示环境。

### 3.5 押金托管与自动结算

**目的**：在合同生效后，租客将押金转入智能合约托管，退房时双方确认后自动结算。
押金归租客所有，暂存合约作为履约担保。仲裁由平台管理员介入。

**仲裁方说明**：

> 演示阶段由平台管理员作为仲裁方。生产环境中，仲裁权可以移交至司法鉴定中心、
> 公证处或第三方仲裁机构，智能合约的执行权限通过多签钱包控制，无需修改代码。

**押金的法律用途**（依据《民法典》租赁合同相关条款）：

| 用途 | 说明 |
|------|------|
| 抵扣租客造成的损坏 | 墙面污损、家具损坏、设备缺失等 |
| 抵扣未结清的水电燃气费 | 退租时发现欠缴费用 |
| 抵扣未付租金 | 租客未缴清最后一个月租金 |
| 抵扣违约金 | 租客提前解约等合同约定的违约行为 |

**押金生命周期（文字步骤）**：

1. **支付押金**：合同双方签署完成后，系统提示租客支付押金。租客通过MetaMask
   调用智能合约`payDeposit(contractId)`，将押金（金额在合同模板中约定）转入合约账户。
   合约锁定该笔资金并发出`DepositPaid`事件。
2. **租期中锁定**：押金在合约中安全托管，不经过平台或任何人之手。租期中不需要补足押金。
3. **退房时结算**：租期结束后，租客发起"申请退还押金"，房东确认或发起扣款：
   - **全额退还**：房东确认无纠纷 → 合约自动将押金全额转回租客钱包。
   - **部分扣除**：房东输入扣款金额和理由 → 租客同意 → 合约按比例分账。
   - **租客不同意扣款** → 进入纠纷状态，管理员介入仲裁。
4. **纠纷仲裁**：管理员在后台查看双方证据（图片、文字说明），手动调用合约的
   `resolveDeposit`函数决定押金归属。

**智能合约新增方法**（在现有CampusTrade.sol中扩展）：

| 方法 | 调用者 | 功能 |
|------|--------|------|
| `payDeposit(contractId)` | 租客 | 支付押金到合约 |
| `requestRefund(contractId)` | 租客 | 退房时发起退款申请 |
| `confirmRefund(contractId)` | 房东 | 同意全额退还押金 |
| `proposeDeduction(contractId, amount)` | 房东 | 提出扣款方案 |
| `acceptDeduction(contractId)` | 租客 | 同意扣款方案 |
| `disputeDeposit(contractId)` | 任一方 | 进入仲裁状态 |
| `resolveDeposit(contractId, toTenant)` | 管理员 | 仲裁裁决(true=退租客) |

**数据库新增字段**（contracts表补充）：

- `deposit_amount`：押金金额（ETH）
- `deposit_status`：押金状态（未支付/已托管/退款中/已退还/已扣除/纠纷中）
- `deposit_tx_hash`：押金支付交易哈希
- `deduction_amount`：扣除金额
- `deduction_reason`：扣款理由

**注意**：押金使用ETH测试币（SepoliaETH）。答辩时说明：
"用ETH测试币验证了押金托管与结算的技术可行性，
生产环境可替换为合规稳定币或数字人民币。"


### 4. 链上验证工具（公开查验）

**目的**：任何用户（无需登录）输入房源ID或合同ID，即可从区块链上查询存证记录，并与本地数据库内容比对，确认是否被篡改。

**实现逻辑（文字步骤）**：

1. 前端提供验证页面，包含输入框、类型选择（房源/合同）、验证按钮。
2. 后端接收请求，根据类型调用智能合约的对应查询函数（如`listings`或`contracts`映射），获取链上存证数据。
3. 同时从本地数据库查询该ID的原始数据。
4. 若链上不存在，返回“链上无此记录”。
5. 若存在，则对比关键字段：
   - 房源：对比图片哈希列表是否完全一致，AI分数是否相同。
   - 合同：对比合同哈希是否一致。
6. 返回验证结果：存证时间、链上数据摘要、对比结果（一致/不一致）、交易哈希链接。
7. 前端展示验证报告，可生成截图作为凭证。

**验证逻辑示例**：假设用户输入合同ID，后端计算出链上合同哈希与本地合同哈希相等，则显示“该合同自签署以来未被修改，存证时间XXXX”。

## 第三部分：支付模块的两种路线（队友二选一）

你们的核心目标是在区块链上记录“租客向房东支付了租金”。支付方式有两条路可选，团队需根据时间和目标协商决定。

### 路线一：纯模拟币（ETH测试币）

**目标**：快速实现链上支付+存证一体化，不涉及任何法币或第三方接口。

**实现逻辑**：
1. 租客和房东安装MetaMask，从Sepolia水龙头领取免费ETH。
2. 在合同签署后，租客进入“支付租金”页面，输入金额（ETH）。
3. 前端调用智能合约的 `payRent` 函数（参数：房东地址、金额）。合约内部：
   - 校验租客余额足够。
   - 执行转账 `payable(房东).transfer(金额)`。
   - 发出 `RentPaid` 事件，记录租客、房东、金额、时间。
4. 支付完成后，前端显示交易哈希，租客和房东均可查证。
5. 若需定期扣款，可在合约中实现预存+定时释放逻辑（如每月自动划转），但需外部触发（可使用Chainlink Automation模拟）。

**优点**：极简单，全链上，演示流畅，无外部依赖。  
**缺点**：不符合国内真实场景，评委可能质疑“脱离国情”。

**答辩话术**：“本项目先用测试币验证了智能合约自动支付与存证的技术可行性。实际商业化时，只需将租客的扣款动作替换为支付宝回调触发上链，代码结构不变。我们已将两种方案都纳入设计文档。”

---

### 路线二：支付宝沙箱模拟（法币回调触发上链）

**目标**：演示“法币支付 → 链上存证”的混合架构，贴近国内真实业务。

**实现逻辑**：
1. 注册支付宝开放平台个人账号，进入沙箱环境，获取虚拟商户信息（APPID、网关、密钥等）。下载沙箱版支付宝APP。
2. 租客在系统内点击“支付宝支付”，后端调用支付宝沙箱的“统一收单下单”接口，生成支付二维码或链接。
3. 租客使用沙箱版支付宝扫码，输入密码（默认为6个1），完成虚拟支付。
4. 支付宝沙箱向开发者配置的后端回调URL发送异步通知（包含订单号、金额、交易状态）。后端验证签名后，确认支付成功。
5. 后端调用智能合约的 `recordPayment` 函数，将订单号、金额、租客地址、房东地址、时间戳上链存证。合约发出 `PaymentRecorded` 事件。
6. 前端轮询或通过WebSocket获取支付结果，显示“支付成功，已上链”。

**注意点**：
- 资金流向：租客的沙箱账户 -> 平台的沙箱商户账户。系统不自动结算给房东，演示时可说明“结算由平台线下完成”。
- 定期扣款：如需模拟每月自动支付，可在后端设置定时任务，每月1日调用支付宝代扣接口（需用户事先授权）。但代扣接口需要签约，沙箱环境可能不支持完整流程，可仅演示单次支付。

**优点**：展示混合架构能力，评委认可度高，符合开题报告的创新点。  
**缺点**：复杂度较高，需理解支付宝接口和签名机制；依赖外部沙箱环境（偶尔不稳定）。

**答辩话术**：“我们使用支付宝官方沙箱环境模拟了法币支付链路，支付成功后通过后端回调触发智能合约存证，实现了支付事实的不可篡改记录。这验证了混合架构的可行性，未来可平滑替换为真实支付宝接口。”

### 路线对比总结表

| 项目 | 路线一（模拟币） | 路线二（支付宝沙箱） |
|------|----------------|---------------------|
| 实现难度 | 低 | 中高 |
| 开发时间 | 2-3天 | 5-7天 |
| 是否需要企业资质 | 否 | 否（个人账号可用沙箱） |
| 是否符合国内场景 | 否（用户不用加密货币） | 是（模拟主流支付） |
| 评委认可度 | 一般 | 高 |
| 区块链作用 | 支付+存证 | 仅存证（支付事实） |
| 定期扣款模拟 | 容易（智能合约定时） | 困难（需代扣签约） |

**团队建议**：如果时间充裕且希望拿高分，选路线二；如果只想快速完成项目、重点放在合同存证和AI检测上，选路线一，并在答辩中说明“支付接口可替换”。

## 第四部分：Web设计方案

### 1. 前端技术栈

- 框架：Vue 3 或 React 18
- UI组件库：Element Plus / Ant Design（快速搭建后台风格）
- 区块链交互：ethers.js + MetaMask（或 WalletConnect）
- HTTP请求：Axios
- 状态管理：Pinia（Vue）或 Redux（React）

### 2. 页面结构（至少包含）

| 路由 | 页面名称 | 主要功能 | 角色限制 |
|------|---------|----------|----------|
| `/login` | 登录页 | 手机号密码登录、注册 | 无 |
| `/register` | 注册页 | 填写信息、绑定钱包 | 无 |
| `/listings` | 房源列表 | 展示所有房源，可筛选 | 任何登录用户 |
| `/listing/:id` | 房源详情 | 显示详情、AI可信度、上链哈希、申请租房按钮 | 租客 |
| `/publish` | 发布房源 | 表单填写、图片上传 | 仅房东 |
| `/contract/:id` | 合同签署 | 展示合同内容、双方签名按钮 | 租客和房东 |
| `/pay/:contractId` | 支付租金 | 选择支付方式（模拟币或支付宝） | 租客 |
| `/verify` | 验证工具 | 输入ID查询链上存证 | 无 |
| `/profile` | 个人中心 | 展示我的房源、我的合同、我的支付记录 | 任何登录用户 |

### 3. 关键交互流程（文字描述）

**房源发布流程**：
- 房东进入`/publish`，填写表单，上传图片。
- 点击“提交”后，前端将图片转为Base64或FormData发送给后端。
- 后端执行AI检测（模拟或真实API），若通过则调用智能合约上链，返回房源ID。
- 前端跳转到房源详情页，显示“发布成功”。

**合同签署流程**：
- 租客在房源详情页点击“申请租房”，前端调用后端创建合同API。
- 后端生成合同草稿，返回合同ID。
- 租客看到合同页面，点击“同意签署”，调用MetaMask签名一条消息（或简单确认）。
- 后端记录租客签署状态。
- 房东登录后，在个人中心看到待签署合同，点击进入，同样操作。
- 双方签署完成后，后端自动调用合约存证。
- 前端显示“合同已生效，下载PDF”。

**支付流程（以路线二为例）**：
- 租客在支付页选择“支付宝支付”，点击“立即支付”。
- 前端调用后端 `/create-payment` 接口，后端返回支付二维码链接。
- 前端展示二维码，并开始轮询后端 `/payment-status` 接口。
- 租客用沙箱版支付宝扫码完成支付。
- 支付宝回调后端，后端上链存证，更新数据库状态。
- 轮询接口返回“支付成功”，前端跳转到成功页。

### 4. 后端API设计（RESTful）

| 接口 | 方法 | 功能 | 说明 |
|------|------|------|------|
| `/api/register` | POST | 用户注册 | 接收手机号、密码、角色 |
| `/api/login` | POST | 用户登录 | 返回token |
| `/api/listings` | GET | 获取房源列表 | 可带筛选条件 |
| `/api/listings` | POST | 发布房源 | 需要房东token |
| `/api/listings/:id` | GET | 获取房源详情 | - |
| `/api/contracts` | POST | 创建合同 | 租客发起 |
| `/api/contracts/:id/sign` | POST | 签署合同 | 双方分别调用 |
| `/api/contracts/:id` | GET | 获取合同详情 | - |
| `/api/payments/create` | POST | 创建支付订单（支付宝） | 返回支付二维码 |
| `/api/payments/callback` | POST | 支付宝异步通知 | 公网可访问 |
| `/api/payments/:id/status` | GET | 查询支付状态 | 前端轮询 |
| `/api/verify/listing/:id` | GET | 验证房源存证 | 无需登录 |
| `/api/verify/contract/:id` | GET | 验证合同存证 | 无需登录 |

### 5. 数据库设计（简略）

- **users**：user_id, phone, pwd_hash, role, wallet_address, created_at
- **listings**：listing_id, landlord_id, address, rent, description, ai_score, image_hashes, tx_hash, status, created_at
- **contracts**：contract_id, listing_id, tenant_id, landlord_id, content_hash, tenant_signed_at, landlord_signed_at, tx_hash, status, pdf_path
- **payments**：payment_id, contract_id, tenant_id, landlord_id, amount, alipay_order_no, onchain_tx_hash, status, created_at

### 6. 部署与环境

- 前端：Vercel / Netlify 或本地演示
- 后端：本地运行或部署到免费云服务（如Render、Railway）
- 数据库：MongoDB Atlas 或 SQLite文件
- 区块链：以太坊Sepolia测试网，使用Infura/Alchemy RPC
- 支付宝沙箱：需将后端回调地址暴露为公网URL（使用ngrok或内网穿透）

## 第六部分：答辩叙事逻辑

1. **背景与痛点**：租赁市场中假房源、合同篡改、押金纠纷频发；国家政策要求平台加强审核，最高法认可区块链存证。
2. **解决方案**：我们基于以太坊测试网开发了“信源链”平台，实现AI辅助房源图片防伪检测并上链、电子合同哈希存证、公开验证工具。
3. **核心创新点**：
   - AI检测结果上链，确保房源真实性可追溯。
   - 电子合同存证遵循《电子签名法》，具备法律效力。
   - 混合支付架构（模拟币或支付宝沙箱），打通链上链下。
4. **系统演示**：播放3-5分钟视频，展示房东注册→发布房源（AI评分88分）→上链存证→租客签约→合同上链→支付租金→验证工具查询成功。
5. **可行性分析**：所有技术成熟，测试网零成本，开发周期可控，法律路径明确。
