# 合同上链字段、签署逻辑与续租设想（讨论稿）

## 1. 范围
- 覆盖合同上链字段、签署状态机、续租设想。
- 暂不考虑押金字段与押金流程。

## 2. 合同上链字段（核心）
- `contractId`
- `listingId`
- `tenantWallet`
- `landlordWallet`
- `contentHash`
- `status`：`draft/tenant_signed/landlord_signed/pending_payment/active/ended/cancelled/expired/disputed`
- `version`（修订合同版本）
- `expiresAt`（签约有效期）
- `tenantSignedAt`、`landlordSignedAt`

## 3. 签署逻辑（建议）
1. `createContract`：先上链创建草案（`contentHash` 固化）。
2. `signAsTenant`：仅租客地址可签。
3. `signAsLandlord`：仅房东地址可签。
4. 双签后进入 `pending_payment`。
5. 支付确认后进入 `active`。

## 4. 取消规则（按当前讨论方向）
- 任一方签署后不可取消。
- 仅双方未签时可取消草案。
- 争议走 `disputed`，不回退成未签。

## 5. 修订与哈希历史
- 修订可走两种：
  - 方案A：新 `contractId`（推荐，审计最清晰）
  - 方案B：同 `contractId` 递增 `version` 并更新 `contentHash`
- 无论 A/B，均发事件保留哈希变更历史。

## 6. 续租设想（不含押金）
- 续租触发条件：合同 `active` 且双方同意。
- 续租动作：生成新版本合同哈希，写链上 `renewContract`。
- 续租后状态：原合同可标记 `ended_by_renewal`，新合同进入 `pending_payment/active`。
- 续租记录事件：`ContractRenewed(oldContractId, newContractId, newVersion, newHash)`。

## 7. 服务器配合职责
- 存 `content_json` 全文、签名原文、审计日志。
- 监听链上事件并回写可视化状态。
- 提供验真接口：重算哈希并与链上比对。
