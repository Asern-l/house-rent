# 合同上链字段、签署逻辑与续租方案

## 1. 范围
- 本文档定义当前版本的合同链上字段、签署锚点、支付生效规则与续租模型。
- 重点说明：谁签、签什么、何时支付、何时生效、续租如何接续。
- 不覆盖押金托管、仲裁、多期支付与完整提前解约链上化。

## 2. 合同链上主字段
当前 `ContractRecord` 建议保留：
- `contractId`
- `listingId`
- `parentContractId`
- `renewalChildContractId`
- `tenant`
- `landlord`
- `contentHash`
- `initialAmountWei`
- `createdAt`
- `startAtMs`
- `endAtMs`
- `status`
- `tenantMessageHash`
- `landlordMessageHash`
- `tenantSignedAt`
- `landlordSignedAt`
- `exists`

说明：
- `parentContractId` 只用于续租合同。
- `renewalChildContractId` 用于从父合同向下定位当前续租链。
- `contentHash` 表示合同正文锚点。

## 3. 合同状态机
当前合同状态：
- `Created`
- `Paid`
- `Active`
- `Completed`
- `Cancelled`

语义：
- `Created`：已创建、未支付。
- `Paid`：续租合同已提前支付，待父合同结束后接续。
- `Active`：合同已正式激活。
- `Completed`：合同已结束。
- `Cancelled`：合同已取消。

重点：
- `status == Active` 不等于“当前一定仍在履约”。
- 当前是否仍有效，必须继续结合 `startAtMs / endAtMs` 判断。

## 4. 签署锚点方案
### 4.1 链上 storage
- `tenantMessageHash`
- `landlordMessageHash`
- `tenantSignedAt`
- `landlordSignedAt`

### 4.2 链上 event
- `ContractSignatureAnchored(contractId, signer, messageHash, role, signature, signedAt)`

### 4.3 链下留档
数据库与合同 PDF 保存：
- 完整签名消息原文
- 完整签名结果 `signature`
- 对应 `messageHash`
- `signer`
- `signedAt`

## 5. 签名消息结构
不要只签 `contentHash`。

建议签结构化消息，至少包含：
- `contractId`
- `contentHash`
- `role`
- `signer`
- `timestamp`
- `deadline`

目的：
- 明确签署语义
- 降低消息替换与重放风险
- 让后端和独立 verifier 都能重建消息并恢复签名地址

## 6. 普通合同主流程
1. 双方完成链下协商。
2. 租客签署消息。
3. 房东签署消息并发起 `createContractRecord(...)`。
4. 合同进入 `Created`。
5. 租客在支付窗口内完成首笔支付。
6. 合同进入 `Active`。
7. 到期后进入 `Completed`，或按懒释放与显式结项规则清理。

## 7. 支付窗口规则
- 采用合约级参数 `paymentWindowMs`。
- 部署脚本由 `PAYMENT_WINDOW_HOURS` 换算传入。
- 支付时校验：
  - `msg.value == initialAmountWei`
  - `block.timestamp * 1000 <= createdAt * 1000 + paymentWindowMs`

## 8. 续租规则
### 8.1 发起人
- 仅租客可在父合同页发起续租。

### 8.2 时间规则
- 子合同 `startAtMs = 父合同 endAtMs`
- 子合同 `endAtMs = 父合同 endAtMs + 续租时长`

### 8.3 条款来源
优先级：
1. 父合同最终条款
2. 房东当前房源默认条款

### 8.4 唯一子合同约束
- 一个父合同同一时间只允许一个子合同。
- 在取消逻辑未完整扩展前，不允许并行第二个续租子合同。

## 9. 续租提前支付模型
- 允许子合同提前支付。
- 提前支付时不切换 `activeContractByListing`。
- 子合同支付成功后进入 `Paid`。
- 父合同结束后，子合同再接续为 `Active`。

## 10. 合同与房源关系
- 合同只需要绑定 `listingId`。
- 房源公开快照历史由房源快照锚点事件负责，不把 `snapshotCid` 直接塞进合同正文锚点。
- 合同独立验真依赖本地 PDF；关联房源的独立验真走链上 `listingId + ListingSnapshotAnchored + IPFS`。

## 11. 当前验真与存储边界
- 合同正文继续由平台数据库主存，不上 IPFS。
- 平台内合同验真优先读取数据库正文、签名消息、签名值与链上锚点。
- 独立合同验真依赖本地保存的 PDF。
- PDF 内嵌的验证标记至少包括：
  - `contractId`
  - `contentHash`
  - `tenantMessageHash`
  - `landlordMessageHash`
  - 签名地址
  - 交易哈希

## 12. 当前方案结论
- 合同链上保存锚点和状态，不保存正文全文。
- 平台内验真看数据库 + 链上锚点。
- 平台外独立验真看本地 PDF + 链上锚点。
- 续租通过 `parentContractId + renewalChildContractId` 表达，不做并行子合同分叉。
