# 合同上链字段、签署逻辑与续租方案

## 1. 范围
- 本文档定义当前版本的合同链上字段、签署锚点、支付生效规则与续约合同模型。
- 重点是把“谁签了、何时可支付、何时算接续生效、如何判断当前有效合同”说清楚。
- 暂不覆盖押金、仲裁、多期支付与提前解约的完整链上化。

## 2. 合同链上主字段
每份合同建议保留以下字段：
- `contractId`
- `listingId`
- `parentContractId`
- `renewalChildContractId`
- `tenant`
- `landlord`
- `contentHash`
- `initialAmountWei`
- `startAtMs`
- `endAtMs`
- `createdAt`
- `tenantMessageHash`
- `landlordMessageHash`
- `tenantSignedAt`
- `landlordSignedAt`
- `status`
- `exists`

说明：
- `parentContractId` 只用于续约合同。
- `renewalChildContractId` 用于从父合同向下查找当前续约链。
- `initialAmountWei` 让合约能独立校验首笔支付金额。
- 支付截止时间不再做每份合同单独字段，而是统一使用合约构造参数 `paymentWindowMs`。

## 3. 合同状态机
当前链上状态集：
- `Created`
- `Paid`
- `Active`
- `Completed`
- `Cancelled`

语义：
- `Created`：已创建/已上链，未支付。
- `Paid`：已支付，但当前只用于“续约已提前支付待接续”。
- `Active`：这份合同曾被正式激活/支付成功过。
- `Completed`：已正式结项。
- `Cancelled`：已取消。

重点：
- `status == Active` 不等于“当前仍有效占用”。
- 当前是否仍有效占用，必须继续结合 `startAtMs` 与 `endAtMs` 判断。

## 4. 签署锚点方案
### 4.1 链上 storage
- `tenantMessageHash`
- `landlordMessageHash`
- `tenantSignedAt`
- `landlordSignedAt`

### 4.2 链上 event
- `ContractSignatureAnchored(contractId, signer, messageHash, role, signature, signedAt)`

### 4.3 链下留档
合同 PDF 与数据库保存：
- 完整签名消息原文
- 完整签名结果 `signature`
- 对应 `messageHash`
- `signer`
- `signedAt`

## 5. 签名消息结构
不要只签 `contentHash`。
建议签结构化消息，至少包含：
- `contractId`
- `contentHash`
- `role`
- `signer`
- `signedAt`
- `deadline`

再对这组内容做哈希，得到：
- `tenantMessageHash`
- `landlordMessageHash`

## 6. 普通合同流程
1. 租客申请并完成链下协商。
2. 租客签署合同消息，并完成 gas 补偿锁仓。
3. 房东签署合同消息并发起 `createContractRecord(...)`。
4. 合同进入链上 `Created`。
5. 租客在支付窗口内完成首笔支付。
6. 合同进入链上 `Active`。

## 7. 支付窗口规则
- 合约级全局参数：`paymentWindowMs`
- 由部署脚本根据 `PAYMENT_WINDOW_HOURS` 换算传入构造函数。
- 支付时合约校验：
  - `msg.value == initialAmountWei`
  - `block.timestamp * 1000 <= createdAt * 1000 + paymentWindowMs`

## 8. 续约合同规则
### 8.1 发起人
- 仅租客可在父合同页发起续约。

### 8.2 时间规则
- 子合同 `startAtMs` 固定等于父合同 `endAtMs`。
- 子合同 `endAtMs = 父合同 endAtMs + 续约时长`。
- 前端只允许编辑续约月数，不允许改起始时间。

### 8.3 条款来源
优先级：
1. 父合同最终条款
2. 房东房源默认条款

### 8.4 唯一子合同约束
- 一个父合同同一时间只允许一个子合同。
- 在取消功能未完整落地前，不允许并行第二个续约子合同。

## 9. 续约提前支付模型
- 允许子合同提前支付。
- 但提前支付时不切换链头映射。
- 子合同支付成功后进入 `Paid` 或等价“已支付待接续”语义。
- 父合同未结束前，当前有效合同仍然是父合同。

## 10. 链头映射与当前有效合同
### 10.1 `activeContractByListing`
- 当前语义：记录这条租赁链的链头合同。
- 不再表示“当前生效合同是谁”。

### 10.2 当前有效合同的判定
从链头合同开始，顺着 `renewalChildContractId` 向下查：
- `Cancelled` / `Completed`：跳过，不算当前有效。
- `Created`：不算当前有效；若仍在支付窗口内，可算“签约中占位”。
- `Paid`：若父合同尚未结束，不算当前有效；若父合同已结束且当前时间落在本合同租期内，则视为已接续生效。
- `Active`：若当前时间位于 `[startAtMs, endAtMs)`，则为当前有效合同。

## 11. 房源前端状态建议
- `签约中`：存在链下签署流程中的合同，或存在 `Created` 且未过支付截止时间的合同。
- `已租出`：存在当前有效合同。
- `可申请`：排除以上两类，且房源本体状态为 `available`。
