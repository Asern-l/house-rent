# 04-ZK匿名星级评价方案评估

## 1. 目标定义

本方案讨论的不是普通租后评价，而是更强约束的一类评价系统：

- 评论内容公开
- 单条星级不公开
- 房东和公众只能看到聚合后的平均星级区间，不知道单条具体评分
- 只有真实已结束合同的租客可以评价
- 租期越长，评价权重越高
- 合同结束后 7 天内允许评价
- 希望评价结果具备链上真实性或至少可验证真实性

这类需求和普通“链下匿名评分”不同，它要求同时满足：

- 真实资格校验
- 单条评分隐藏
- 聚合结果可信
- 评论可公开阅读

如果要在链上实现以上目标，ZK 是可行方向，但工程复杂度很高。

## 2. 为什么普通方案不够

如果不用 ZK，只做普通链上或链下评分，常见方案有两类：

### 2.1 明文评分

链上或数据库直接记录：

- `rating`
- `comment`
- `contractId`
- `raterRole`

问题：

- 单条评分直接公开
- 无法满足“只看区间、不看单条”的要求

### 2.2 哈希承诺 / commit-reveal

链上先存：

- `commitHash = keccak256(rating, commentHash, salt, contractId, role)`

之后再 reveal：

- `rating`
- `commentHash`
- `salt`

优点：

- 到期前可隐藏评分
- 可证明评分未被事后篡改

问题：

- reveal 后单条评分仍公开
- 仍无法满足“评论公开但单条星级不公开”

因此，如果目标是“公开评论、隐藏单条评分、只公开聚合区间”，普通哈希方案不够，需要更强的隐私证明或可信聚合机制。

## 3. ZK 在这个需求里的正确使用方式

ZK 在这里不应当用于“把评论全文加密”。

更合理的用法是：

- 公开评论明文
- 不公开单条评分值
- 用 ZK 证明评价资格与评分合法性
- 链上只接受合法评分对聚合统计的贡献

也就是说，ZK 保护的是：

- 谁在评价
- 这条评价对应哪份合同
- 这条评价的具体星级

而不是保护评论文本。

## 4. 目标协议结构

### 4.1 公开部分

公开展示给所有人：

- 评论明文
- 评价时间
- 房源 ID
- “真实租客评价”标记
- 聚合后的星级区间

不公开：

- 单条具体评分
- 评价者具体身份
- 评价绑定的具体合同 ID

### 4.2 私密部分

私下保留或通过证明隐藏：

- 单条 `rating`
- `contractId`
- `tenant wallet / user id`
- `leaseMonths / weight`

### 4.3 需要证明的事实

ZK 电路至少要证明：

1. 评价者对应一份真实存在的合同
2. 该合同已结束
3. 当前时间处于 `endAt + 7 days` 的评价窗口内
4. 评价者是合同合法租客
5. 该租客尚未对该合同重复评价
6. 评分值在合法范围内，例如 `1..5`
7. 权重来自真实租期，而不是评价者随便填写

## 5. 链上状态设计方向

### 5.1 每条评价不存明文 rating

链上不存：

- `rating = 4`
- `rating = 2`

链上只存：

- 已使用的 `nullifier`
- 聚合总权重
- 聚合加权总分的承诺值或可验证更新结果
- 评论哈希或评论索引引用

### 5.2 评论明文的存储建议

评论全文不建议直接上链。

更合理的结构是：

- 链上：`commentHash`
- 链下：评论明文
- 前端展示评论时，可用 `commentHash` 对评论做防篡改校验

原因：

- 评论全文上链 gas 成本高
- 文本不可撤回，隐私风险大
- ZK 也不需要保护评论明文本身

## 6. 为什么还要显示“区间”而不是精确平均分

如果只隐藏单条评分，但公开精确平均分，例如：

- 当前平均分 `4.37`
- 新增一条评价后变成 `3.91`

在样本较少时，房东很容易反推出新评价大致是多少分。

因此更合理的是只显示区间，例如：

- `4.0 - 4.4`
- `4.5 - 4.7`
- `3.5 - 3.9`

同时建议加入最小样本阈值：

- 少于 3 条正式评价时，不展示区间
- 少于 5 条时，只展示粗粒度区间或“样本较少”提示

否则匿名性仍然会被平均值变化击穿。

## 7. 权重设计

你要求“租期越长，星级权重越大”，这是合理的。

建议的权重来源：

- `weight = leaseMonths`
- 或 `weight = ceil(leaseDays / 30)`

这样一来，聚合统计变为：

- `weightedScoreSum += rating * weight`
- `weightSum += weight`

展示时再根据：

- `weightedScoreSum / weightSum`

映射到区间。

关键点：

- 权重必须来自合同真实租期
- 不允许评价者自行传入权重
- ZK 电路中要证明该权重来自真实已结束合同

## 8. 真正的工程复杂度在哪里

ZK 不是“多写一个合约函数”这么简单，而是新增一整套密码学子系统。

### 8.1 协议设计复杂

要先明确：

- 哪些数据公开
- 哪些数据隐藏
- 聚合统计如何更新
- 如何防止重复评价
- 如何避免小样本反推

这一步本身就比写代码更重要。

### 8.2 电路开发复杂

需要专门写 ZK 电路，证明：

- 合同资格
- 评分合法区间
- 时间窗口合法
- 权重合法
- nullifier 未使用

这通常要用：

- Circom
- Noir
- Halo2 或类似系统

这不是现有 Solidity/React/Node 技能栈里的自然延伸。

### 8.3 证明生成复杂

每次评价时都要生成 proof：

- 浏览器端生成 proof，可能较慢
- 后端代生成，又会引入新的信任边界
- 调试成本高

### 8.4 链上 verifier 集成复杂

需要新增：

- verifier 合约
- nullifier 存储
- 聚合状态更新逻辑
- 防重复与防重放逻辑

### 8.5 聚合模型复杂

这类系统最难的点之一不是“证明我有资格评价”，而是：

- 在不公开单条评分的前提下，如何安全地维护聚合统计
- 且保证聚合结果可信

如果这一步设计不好，系统虽然用了 ZK，但产品目标仍然不能真正成立。

## 9. 复杂度结论

相对普通“真实租客评分”模块，ZK 版大致可以认为是：

- 开发复杂度：`5x ~ 10x`
- 如果团队此前没有 ZK 经验，实际主观成本往往更接近 `10x+`

原因不是代码量本身，而是：

- 新协议
- 新工具链
- 新调试方法
- 新链上验证逻辑
- 新产品隐私边界

## 10. 当前项目是否建议立即做

### 10.1 不建议立即做完整 ZK 版

对当前项目阶段，更现实的判断是：

- ZK 版方向成立
- 但不是高性价比的第一步
- 直接一次做到位，风险和工期都偏大

### 10.2 更合理的分阶段路线

#### 第一阶段：最小可实现版

先不做 ZK，只做产品语义验证：

- 只有真实已结束合同的租客可评
- 评论公开
- 单条星级不展示
- 后台保存真实评分
- 前台只显示加权平均分区间
- 少量样本不展示区间
- 权重按租期长度
- 评价窗口为合同结束后 7 天

这能先验证以下问题：

- 产品语义是否成立
- 用户是否能理解“评论公开、评分不公开”
- 区间展示是否足够好用
- 租期权重是否符合预期

#### 第二阶段：ZK 增强版

在第一阶段产品语义跑通后，再升级为：

- 评价资格用 ZK 证明
- 单条评分不上公开链
- 聚合统计通过链上 proof 更新
- nullifier 防止重复评价

这样复杂度才可控。

## 11. 推荐结论

如果目标是“做一个概念上很强、真正隐私友好的链上评价系统”，ZK 方向是成立的。

但如果目标是“当前项目内尽快落地可演示版本”，建议采用：

1. 先做无 ZK 的最小版
2. 保留后续升级到 ZK 的字段和协议空间
3. 等产品语义验证通过后，再决定是否投入 ZK 实现

最终判断：

- **ZK 不是不行**
- **而是当前阶段直接上完整 ZK 版，复杂度明显过高**
- **最佳路线是先做普通版，再预留 ZK 升级路径**
