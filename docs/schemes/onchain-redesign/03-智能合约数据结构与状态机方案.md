# 智能合约数据结构与状态机方案

## 1. 文档目标
- 汇总当前 `RentalChain.sol` 的核心数据结构与状态机语义。
- 说明房源、合同、快照、评论、支付、续租之间的链上边界。
- 为后端回写校验、独立 verifier、文档答辩提供统一口径。

## 2. 当前链上模块
当前合约主要承载 4 类能力：
1. 房源记录与状态切换
2. 合同记录、签署锚点、支付生效与续租接续
3. 房源公开快照历史锚点
4. 房源反馈与租后评价事件锚点

## 3. 角色与权限
- `landlord`
  - 创建房源
  - 更新房源条款
  - 修改房源状态
  - 创建合同记录
- `tenant`
  - 支付首笔租金
  - 对已结束合同发起租后评价
- `landlord` 或 `tenant`
  - 取消未生效合同
  - 释放 gas 补偿授权
- 任意地址
  - 对已到期合同调用 `completeExpiredContract(...)`

## 4. 房源数据结构
### 4.1 主记录 `ListingRecord`
核心字段：
- `listingId`
- `landlord`
- `contentHash`
- `rentAmountWei`
- `minLeaseMonths`
- `imageRootHash`
- `status`
- `version`
- `nonce`
- `createdAt`
- `updatedAt`
- `exists`

### 4.2 主记录语义
- `contentHash` 只锚定稳定字段。
- `rentAmountWei`、`minLeaseMonths`、`imageRootHash` 属于可变链上字段。
- 房源是否“已租出”不直接由 `status` 表达。

## 5. 房源快照历史
### 5.1 事件
- `ListingSnapshotAnchored(listingId, version, contentHash, snapshotHash, snapshotCid, blockTime)`

### 5.2 作用
- 保存每一版公开快照的 `snapshotCid`
- 保存该快照文件哈希 `snapshotHash`
- 记录它对应的 `version` 与 `contentHash`

### 5.3 当前用途
- 平台数据库历史日志的二次核对
- 独立 verifier 自动读取最新快照
- 独立 verifier 按时间追溯历史版本

## 6. 合同数据结构
### 6.1 主记录 `ContractRecord`
核心字段：
- `contractId`
- `listingId`
- `parentContractId`
- `renewalChildContractId`
- `tenant`
- `landlord`
- `contentHash`
- `initialAmountWei`
- `createdAt`
- `startAtMs`
- `endAtMs`
- `status`
- `tenantMessageHash`
- `landlordMessageHash`
- `tenantSignedAt`
- `landlordSignedAt`
- `exists`

### 6.2 合同存储边界
- 合同正文不上链
- 合同正文不上 IPFS
- 合同正文继续由平台数据库主存
- 独立合同验真依赖本地 PDF

## 7. 评论与评价数据结构
### 7.1 房源反馈
- 方法：`submitListingFeedback(listingId, feedbackType, commentHash, commentCid)`
- 事件：`ListingFeedbackSubmitted(listingId, sender, feedbackType, commentHash, commentCid, createdAt)`

### 7.2 租后评价
- 方法：`submitRentalReview(contractId, commentHash, rating, commentCid)`
- 事件：`RentalReviewSubmitted(contractId, listingId, tenant, rating, commentHash, commentCid, ratedAt)`

### 7.3 哈希语义
- `commentCid` 用于定位 IPFS 正文
- `commentHash` 对应规则是 `keccak256(commentText)`
- 独立 verifier 读取 IPFS JSON 后，应取其中 `commentText` 重算，而不是对整份 JSON 文本做哈希

## 8. 状态枚举
### 8.1 房源状态 `ListingStatus`
- `Active`
- `Offline`
- `Closed`

语义：
- `Active`：可展示、可申请、可签约
- `Offline`：主动下架
- `Closed`：永久关闭

### 8.2 合同状态 `ContractStatus`
- `None`
- `Created`
- `Paid`
- `Active`
- `Completed`
- `Cancelled`

语义：
- `Created`：已创建、未支付
- `Paid`：续租合同已提前支付，待父合同结束后接续
- `Active`：已正式激活
- `Completed`：已结束
- `Cancelled`：已取消

重点：
- `status == Active` 不等于当前一定仍有效占用
- 当前是否仍有效，必须继续结合 `startAtMs / endAtMs`

## 9. 房源占用索引
- `activeContractByListing[listingId]`

当前语义：
- 更接近这条租赁链的链头索引
- 不是前端可直接认定“当前有效合同是谁”的唯一真相

当前有效合同判定仍需结合：
- 合同链
- `renewalChildContractId`
- `status`
- `startAtMs / endAtMs`

## 11. 当前主流程状态机
### 11.1 普通合同
1. 房东创建合同记录 -> `Created`
2. 租客在支付窗口内支付 -> `Active`
3. 到期或显式结项 -> `Completed`
4. 未生效前取消 -> `Cancelled`

### 11.2 续租合同
1. 基于父合同发起续租
2. 子合同创建 -> `Created`
3. 父合同结束前提前支付 -> `Paid`
4. 父合同结束后接续 -> `Active`
5. 到期后 -> `Completed`

## 12. 与独立 verifier 的关系
### 12.1 合同独立验真
- 输入：本地 PDF
- 读取：链上合同记录、签名锚点
- 不依赖平台账号系统

### 12.2 房源独立验真
- 输入：`listingId`
- 默认自动读取该房源最新 `ListingSnapshotAnchored`
- 可按时间戳追溯历史版本
- 图片只按 `imageCids` 走 IPFS gateway 读取

### 12.3 评论独立验真
- 读取链上 `commentCid`
- 从 IPFS 取正文
- 对 `commentText` 做 `keccak256(...)`
- 与链上 `commentHash` 比较

## 13. 当前实现边界
- 不做房源全文直接上链
- 不做合同正文 IPFS 化
- 不做链上平均分聚合
- 不做匿名评论或 ZK 评价
- 不把平台历史页面直接建立在链上事件扫描之上

## 14. 当前方案结论
- 房源主记录负责当前状态和稳定锚点
- 房源快照历史通过 `ListingSnapshotAnchored` 保留
- 合同正文继续数据库主存，本地 PDF 负责独立合同验真
- 评论与评价通过 `commentCid + commentHash` 完成独立材料级校验
