# 房源上链字段与逻辑方案

## 1. 目标与意义
- 目标 1：让“房源是否存在、谁创建、何时变更、变更到哪一版”可被链上独立验证。
- 目标 2：让房源链上状态只表达房源本身是否可见，不再把“是否已出租”这种派生占用状态重复存入房源记录。
- 目标 3：把房源占用关系交给合同状态与 `activeContractByListing` 映射，避免房源状态和合同状态双写失真。

设计意义：
- 对用户：房源本体和房源占用是两套清晰语义，验真更直观。
- 对平台：减少“房源显示 rented，但合同已结束”这类状态不一致。
- 对开发：房源状态机和合同状态机边界明确，续约扩展更自然。

---

## 2. 设计原则
- 最小上链：只上链“锚点与状态”，不把全文描述和原图直接上链。
- 房源与占用分离：房源链上状态只表示 `active / offline / closed`；占用由生效合同单独表达。
- 权限最小化：仅房东绑定钱包可操作该房源链上状态与条款。
- 幂等可恢复：后端落库前必须校验链上交易；失败可重试，不产生重复业务结果。
- 可审计：每次链上成功都写本地操作日志，绑定快照哈希与链上版本/nonce。

---

## 3. 房源上链字段
房源在链上由以下核心字段表达：
- `listingId`：平台房源ID。
- `landlord`：房东钱包地址。
- `contentHash`：房源核心内容哈希。
- `rentAmountWei`：链上租金（wei）。
- `minLeaseMonths`：最少租期（月）。
- `imageRootHash`：图片哈希根。
- `status`：房源状态枚举，仅保留 `Active / Offline / Closed`。
- `version`：版本号；每次条款或状态更新递增。
- `nonce`：操作序号；每次条款或状态更新递增。
- `createdAt`、`updatedAt`：链上区块时间。
- `exists`：记录存在标记。

说明：
- `contentHash` 作为房源正文锚点，不通过改价/改图流程修改。
- `rentAmountWei`、`minLeaseMonths`、`imageRootHash` 允许链上更新。
- 房源链上不再维护 `rented`。

---

## 4. 房源链上状态模型
- `active`：房源可展示、可签约。
- `offline`：房东主动下架，不可发起新签约。
- `closed`：房源永久关闭，不可恢复。

约束：
- `closed` 为终态。
- `active / offline` 允许改价、改最少租期、改图片根哈希。
- 房源是否“已出租”，不由 `status` 表达，而由合同映射决定。

---

## 5. 房源占用模型
新增链上映射：
- `activeContractByListing[listingId] = contractId`

语义：
- 若为空字符串，说明当前没有生效合同占用该房源。
- 若非空，说明链上当前登记有一份最近一次被激活的合同。

效果：
- 新合同创建前，先检查该房源是否已有生效合同。
- 首笔支付成功后，才真正把该合同登记为当前占用合同。
- 合同结束后，再释放该映射。

重点说明：
- `activeContractByListing` 是“当前登记中的占用合同索引”，不是脱离时间语义的最终真相。
- `status == Active` 不等于“当前仍有效占用”。
- 当前是否仍有效占用，应按“合同已激活，且当前时间尚未超过 `endAtMs`”判断。
- 因此，即使某合同链上 `status` 还停在 `Active`，只要它已经过了 `endAtMs`，前端和业务判断都不应再把它视为当前仍在租。

---

## 6. 与合同的协同规则
### 6.1 创建合同时
- 房东上链创建合同前，先检查房源链上状态必须为 `Active`。
- 若 `activeContractByListing[listingId]` 为空：允许正常创建合同。
- 若不为空：
  - 默认拒绝创建普通合同；
  - 若新合同明确引用当前生效合同为 `parentContractId`，则允许作为续约合同创建，但此时仅创建 `Created` 记录，不立即占用。

### 6.2 支付激活时
- 支付前再次检查是否已有未释放的生效合同。
- 若没有，则当前合同从 `Created -> Active`，并写入 `activeContractByListing[listingId] = contractId`。

### 6.3 到期释放时
- 合约提供 `completeExpiredContract(contractId)`。
- 任何人都可在合同到期后调用。
- 执行后：
  - 合同状态变为 `Completed`
  - 若它正占用该房源，则清空 `activeContractByListing[listingId]`
- 同时，在创建合同/支付合同前也会自动尝试懒释放已过期的旧生效合同，避免历史占用卡死后续流程。

---

## 7. contentHash 的边界
当前纳入 `contentHash` 的字段：
- `title`
- `description`
- `address`
- `district`
- `bedrooms`
- `livingrooms`
- `bathrooms`
- `area`
- `listingId`
- `landlordWallet`
- `imageHashes`

当前不纳入 `contentHash`、而是独立链上维护的字段：
- `rentAmountWei`
- `minLeaseMonths`
- `imageRootHash`

---

## 8. 发布与修改流程
### 8.1 发布
1. 后端 `prepare-create` 生成 `draft` 与 `chainAnchor`。
2. 前端钱包发起 `createListing(...)`。
3. 交易确认后，后端 `commit-create` 校验：
   - 目标合约地址
   - 发送者地址
   - 事件名与参数
4. 校验通过后写入：
   - `listings`
   - `listing_chain_operations`
   - `listing_operation_logs`

### 8.2 条款更新
1. 后端 `terms/prepare` 生成更新锚点。
2. 前端钱包发起 `updateListingTerms(...)`。
3. 后端 `terms/commit` 校验链上回执与事件。
4. 校验通过后更新本地租金、最少租期、图片等字段，并写历史绑定日志。

### 8.3 状态更新
1. 后端 `status/prepare` 生成目标状态与约束校验。
2. 前端钱包发起 `setListingStatus(...)`。
3. 后端 `status/commit` 校验链上事件与 `version/nonce`。
4. 校验通过后更新本地状态并写历史日志。

---

## 9. 当前实现结论
- 房源链上不再使用 `rented` 状态。
- 房源是否被占用，改为由生效合同映射 `activeContractByListing` 表达。
- 创建合同与支付激活都会检查房源当前是否已被生效合同占用。
- 已过期的生效合同可通过显式调用 `completeExpiredContract(...)` 或后续业务操作时的懒释放自动清理。
