# 房源上链字段与逻辑方案

## 1. 文档目标
- 说明房源在链上到底保存什么、不保存什么。
- 说明房源当前状态、房源占用状态、房源公开快照三者的边界。
- 统一平台回写、独立 verifier、答辩说明的口径。

## 2. 设计结论
当前房源链上采用“三层表达”：
1. 房源主记录：表达当前房源状态与关键锚点。
2. 房源快照锚点：表达某一版公开材料对应的 `snapshotCid / snapshotHash`。
3. 合同链索引：表达房源是否被当前有效合同占用。

这意味着：
- 房源主记录不承载公开材料全文。
- 图片和公开快照走 IPFS。
- 房源是否“已租出”不直接由房源主状态表达。

## 3. 房源主记录字段
当前 `ListingRecord` 的核心字段：
- `listingId`
- `landlord`
- `contentHash`
- `rentAmountWei`
- `minLeaseMonths`
- `imageRootHash`
- `status`
- `version`
- `nonce`
- `createdAt`
- `updatedAt`
- `exists`

这些字段分成两类：

### 3.1 稳定锚点字段
- `listingId`
- `landlord`
- `contentHash`

### 3.2 当前可变字段
- `rentAmountWei`
- `minLeaseMonths`
- `imageRootHash`
- `status`
- `version`
- `nonce`

## 4. contentHash 的边界
当前纳入 `contentHash` 的稳定字段是：
- `listingId`
- `landlordWallet`
- `title`
- `description`
- `address`
- `district`
- `bedrooms`
- `livingrooms`
- `bathrooms`
- `area`

当前不纳入 `contentHash` 的字段是：
- `rentAmountWei`
- `minLeaseMonths`
- `imageRootHash`
- `imageHashes`
- `snapshotHash`
- `snapshotCid`
- `status`

设计理由：
- `rentAmountWei`、`minLeaseMonths`、图片材料允许在未被阻塞时正常更新。
- 如果这些字段进入 `contentHash`，普通改价、改图会被误判为房源正文被篡改。
- 因此它们应由“当前链上字段 + 房源快照材料层”共同完成验真。

## 5. 房源状态模型
当前房源状态枚举：
- `Active`
- `Offline`
- `Closed`

语义：
- `Active`：可展示、可申请、可签约。
- `Offline`：房东主动下架。
- `Closed`：永久关闭。

约束：
- `Closed` 为终态。
- `Active / Offline` 允许切换。
- 房源是否“当前已出租”不由 `status` 直接表达。

## 6. 房源占用模型
链上映射：
- `activeContractByListing[listingId]`

当前语义：
- 它更接近这条租赁链的链头索引。
- 它不是前端可直接认定“当前有效合同是谁”的唯一真相。

当前是否仍被占用，仍需结合：
- 合同链
- `renewalChildContractId`
- `status`
- `startAtMs / endAtMs`

## 7. 房源公开快照
房源公开快照是链下材料，不直接把 JSON 全文上链。

快照中通常包含：
- `listingId`
- 标题、描述、地址、区域
- 户型与面积
- `rentAmount`
- `minLeaseMonths`
- `imageCids`
- `imageHashes`
- 当前 `contentHash`

这份快照整体上 IPFS，得到：
- `snapshotCid`
- `snapshotHash`

## 8. 房源快照链上锚点
每次发布或修改房源公开内容时，链上发出：

`ListingSnapshotAnchored(listingId, version, contentHash, snapshotHash, snapshotCid, blockTime)`

它的作用是：
- 固定这一版公开快照的 IPFS 地址
- 固定这一版快照文件哈希
- 将该版快照与当时的 `version`、`contentHash` 绑定

## 9. 图片材料规则
图片不逐张直接上链。

图片验真采用：
- 快照中的 `imageCids`
- 快照中的 `imageHashes`
- 链上的 `imageRootHash`

独立 verifier 当前口径：
- 图片只按 `imageCids` 走 IPFS gateway 读取
- 不再把平台 `/uploads/...` 路径当作正式验真来源

## 10. 发布与修改流程

### 10.1 发布房源
1. 后端准备草稿与链上锚点。
2. 后端先写入房源公开快照到 IPFS。
3. 前端钱包发起 `createListing(...)`。
4. 链上成功后，后端校验回执与 `ListingSnapshotAnchored`。
5. 校验通过后再写数据库和操作日志。

### 10.2 修改房源条款
1. 后端准备新的可变字段与新的公开快照。
2. 新快照写入 IPFS，生成新的 `snapshotCid / snapshotHash`。
3. 前端钱包发起 `updateListingTerms(...)`。
4. 后端校验链上版本、事件与快照锚点。

### 10.3 修改房源状态
1. 前端钱包发起 `setListingStatus(...)`。
2. 后端校验 `ListingStatusChanged`、`version`、`nonce`。
3. 更新数据库当前状态和历史日志。

## 11. 独立验真口径
独立 verifier 对房源的正式验真应分成两层：

### 11.1 当前链上房源验真
核对：
- `listingId`
- `contentHash`
- `rentAmountWei`
- `minLeaseMonths`
- `imageRootHash`
- `status`

### 11.2 快照材料验真
核对：
- 自动读取或手动指定 `snapshotCid`
- 从 IPFS 取回公开快照
- 校验 `snapshotHash`
- 校验快照中的当前字段与链上是否一致
- 按 `imageCids` 读取图片材料

## 12. 当前方案结论
- 房源主记录负责当前状态和稳定锚点。
- 房源公开材料通过 `snapshotCid / snapshotHash` 固定版本。
- 图片材料不进入 `contentHash`，由 `imageRootHash + imageCids + snapshotHash` 完成验真。
- 房源是否被占用由合同链和时间语义决定，不再由房源状态直接表达。
