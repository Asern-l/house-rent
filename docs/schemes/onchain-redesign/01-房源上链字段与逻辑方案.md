# 房源上链字段与逻辑方案

## 1. 目标
- 让房源的创建、修改、状态切换、版本演进可被链上独立验证。
- 让房源“本体状态”和“当前是否被合同占用”分离表达，避免双写失真。
- 让公开验真材料走 IPFS，链上只保留锚点和索引。

## 2. 设计原则
- 最小上链：不上链房源全文、图片原文件、评论明文。
- 锚点优先：链上保存 `contentHash`、`imageRootHash`、`snapshotHash`、`snapshotCid` 等锚点信息。
- 版本可追：每次发布或修改都能对应到一个链上版本和一份公开快照。
- 状态分层：房源本体状态只描述可见性与可操作性，不直接表达“已租出”。

## 3. 房源链上主字段
当前 `ListingRecord` 的核心字段：
- `listingId`
- `landlord`
- `contentHash`
- `rentAmountWei`
- `minLeaseMonths`
- `imageRootHash`
- `status`
- `version`
- `nonce`
- `createdAt`
- `updatedAt`
- `exists`

说明：
- `contentHash` 只锚定稳定字段。
- `rentAmountWei`、`minLeaseMonths`、`imageRootHash` 属于当前可变展示字段，单独上链维护。
- `version` 与 `nonce` 用于约束链上修改顺序和回写一致性。

## 4. 房源状态模型
当前房源状态枚举：
- `Active`
- `Offline`
- `Closed`

语义：
- `Active`：可展示、可申请、可签约。
- `Offline`：房东主动下架，不可发起新签约。
- `Closed`：永久关闭，不再恢复。

约束：
- `Closed` 为终态。
- `Active / Offline` 可切换。
- 房源是否“当前已出租”，不由 `status` 表达，而由合同链与时间语义决定。

## 5. 房源占用模型
链上映射：
- `activeContractByListing[listingId]`

当前语义：
- 它更适合作为这条租赁链的链头索引，而不是前端直接认定“当前有效合同”的唯一真相。
- 当前是否仍然有效占用，仍需结合合同链、`status`、`startAtMs`、`endAtMs` 继续判断。

## 6. contentHash 的边界
当前纳入 `contentHash` 的稳定字段：
- `listingId`
- `landlordWallet`
- `title`
- `description`
- `address`
- `district`
- `bedrooms`
- `livingrooms`
- `bathrooms`
- `area`

当前不纳入 `contentHash` 的字段：
- `rentAmountWei`
- `minLeaseMonths`
- `imageRootHash`
- `imageHashes`
- `snapshotHash`
- `snapshotCid`
- `status`

这样做的原因：
- `rentAmountWei`、`minLeaseMonths`、图片材料允许在未被阻塞的情况下正常修改。
- 这些字段若进入 `contentHash`，普通改价、改图会被误判为“正文锚点变化”。
- 它们应改由当前链上字段和快照材料层共同验真。

## 7. 图片与公开快照
图片与公开房源材料采用两层结构：
1. 图片文件走 IPFS，快照内保存 `imageCids / imageHashes`。
2. 快照整体再上 IPFS，形成 `snapshotCid / snapshotHash`。

链上不逐张存图片 CID，而是通过：
- `imageRootHash`
- `ListingSnapshotAnchored(...)`
完成图片集合和快照版本的锚定。

## 8. 房源快照历史锚点
每次发布或修改房源公开内容时，链上额外发出：

`ListingSnapshotAnchored(listingId, version, contentHash, snapshotHash, snapshotCid, blockTime)`

作用：
- 固定这一版快照的 `snapshotCid`
- 固定这一版快照文件哈希 `snapshotHash`
- 记录它对应的 `version` 和 `contentHash`

这条事件主要服务于：
- 独立 verifier 自动读取最新快照
- 独立 verifier 按时间追溯历史版本
- 平台对数据库历史记录做二次核对

## 9. 发布与修改流程
### 9.1 发布房源
1. 后端准备草稿与链上锚点。
2. 后端先写入房源公开快照到 IPFS。
3. 前端钱包发起 `createListing(...)`。
4. 链上成功后，后端校验回执与 `ListingSnapshotAnchored` 事件。
5. 校验通过后写数据库、操作台账、历史日志。

### 9.2 修改房源条款
1. 后端准备新的可变字段与新的公开快照。
2. 新快照写入 IPFS，生成新的 `snapshotCid / snapshotHash`。
3. 前端钱包发起 `updateListingTerms(...)`。
4. 后端校验链上事件、版本号、快照锚点，并更新数据库记录。

### 9.3 修改房源状态
1. 前端钱包发起 `setListingStatus(...)`。
2. 后端校验 `ListingStatusChanged`、`version`、`nonce`。
3. 更新数据库当前状态与历史日志。

## 10. 独立验真口径
房源独立验真应分两层：
1. 当前链上房源验真
- `listingId`
- `contentHash`
- `rentAmountWei`
- `minLeaseMonths`
- `imageRootHash`
- `status`

2. 快照材料验真
- 自动读取或手动指定 `snapshotCid`
- 从 IPFS 取回公开快照
- 校验 `snapshotHash`
- 校验快照中的当前字段与链上是否一致
- 图片只按 `imageCids` 走 IPFS gateway 读取

## 11. 当前方案结论
- 房源主记录负责当前状态与稳定锚点。
- 快照历史通过 `ListingSnapshotAnchored` 保留。
- 图片材料不进入 `contentHash`，由 `imageRootHash + imageCids + snapshotHash` 完成验真。
- 房源是否被占用不再由房源状态直接表达，而由合同链和时间语义决定。
