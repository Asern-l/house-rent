# 租后评价与房源反馈方案

## 目标
- 区分“房源反馈”和“租后评价”两类内容。
- 房源反馈公开展示，链上锚定 `feedbackType + commentHash`，不计入正式星级。
- 租后评价公开展示，单向仅租客提交，链上锚定 `rating + commentHash`。
- 房源正式平均分按租期月数加权，至少 3 条真实租客评价后才公开。

## 一、两类评价的边界

### 1. 房源反馈
- 入口：
  - 房源详情页摘要入口
  - 独立房源评论页 `ListingReviewsPage`
- 对象：登录用户、且不是该房源房东。
- 作用：补充“与描述不符、图片过旧、位置噪音大、沟通体验一般”等看房反馈。
- 存储：
  - 链上：`feedbackType + commentHash + listingId + sender + createdAt`
  - 数据库：公开评论明文、反馈类型、钱包地址、交易哈希
- 展示：公开展示反馈类型、文本、时间和钱包地址。
- 计分：不参与正式星级平均分。
- 第一版限制：允许重复提交，是否灌水由链上 gas 成本自然约束。

### 2. 租后评价
- 入口：合同详情页底部。
- 对象：真实租客，且仅合同租客本人可提交。
- 方向：单向，仅租客评价，不再做双向互评。
- 存储：
  - 链上：`rating + commentHash + contractId + listingId + tenant + ratedAt`
  - 数据库：公开评论明文、星级、权重、交易哈希
- 展示：公开钱包地址、公开星级、公开评论、公开时间。
- 计分：参与房源正式平均分。

## 二、租后评价资格规则
- 一份合同最多评价一次。
- 仅合同租客本人可以发起链上评价交易。
- 合同必须是正常履约结束语义：后端按 `status in ('active','ended')` 且租期已过判断；链上按 `Active/Completed + endAtMs` 判断。
- 评价窗口：合同结束后 30 天内。
- 续租不影响老合同评价：每份合同独立评价、独立计权。
- `pending / tenant_signed / pending_payment / expired / cancelled` 这类未正常完成的合同不开放租后评价。

## 三、正式平均分规则
- 仅使用“租后评价”参与平均分。
- 权重按合同租期月数计算：`weight = leaseMonths`，最小为 1。
- 计算方式：
  - `weightedScoreSum += rating * weight`
  - `weightSum += weight`
  - `weightedAverage = weightedScoreSum / weightSum`
- 至少 3 条真实租客评价后，才公开平均分。
- 在不足 3 条前：
  - 评价列表可公开展示
  - 平均分显示为“样本不足”

## 四、链上设计

### 1. 合约函数
- `submitRentalReview(string contractId, bytes32 commentHash, uint8 rating)`

### 2. 合约校验
- `contractId` 非空
- `commentHash != 0`
- `rating` 在 `1..5`
- `msg.sender == contract.tenant`
- 合同存在
- 合同状态为 `Active` 或 `Completed`
- 当前时间已超过 `endAtMs`
- 当前时间未超过 `endAtMs + 30 days`
- 该合同尚未提交过评价

### 3. 链上事件
- `RentalReviewSubmitted(contractId, listingId, tenant, rating, commentHash, ratedAt)`

### 4. 链上不存什么
- 不存评论明文
- 不做匿名隐藏星级
- 不做零知识证明
- 不做链上加权平均聚合

## 五、后端落地

### 1. 新表：`listing_feedbacks`
字段：
- `id`
- `listing_id`
- `author_id`
- `author_role`
- `author_wallet`
- `feedback_type`
- `comment_text`
- `comment_hash`
- `tx_hash`
- `chain_env`
- `created_at`

### 2. 新表：`contract_reviews`
字段：
- `id`
- `contract_id`
- `listing_id`
- `tenant_id`
- `tenant_wallet`
- `rating`
- `weight`
- `comment_text`
- `comment_hash`
- `tx_hash`
- `chain_env`
- `created_at`

### 3. 路由
- `POST /listings/:id/feedbacks`
  - 房源反馈链上回写
- `POST /contracts/:id/review/onchain`
  - 租客链上评价确认回写
- `GET /listings/:id`
  - 返回 `feedbacks`、`tenant_reviews`、`review_summary`
- `GET /contracts/:id`
  - 返回 `rental_review`、`review_state`

### 4. 回写校验
后端在写入 `contract_reviews` 前校验：
- receipt 成功
- 方法名为 `submitRentalReview`
- `tx.from` 与合同租客钱包一致
- `contractId` 一致
- `rating` 一致
- `commentHash` 与提交明文重新计算结果一致
- 存在 `RentalReviewSubmitted` 事件且参数一致

后端在写入 `listing_feedbacks` 前校验：
- receipt 成功
- 方法名为 `submitListingFeedback`
- `tx.from` 与当前登录钱包一致
- `listingId` 一致
- `feedbackType` 一致
- `commentHash` 与提交明文重新计算结果一致
- 存在 `ListingFeedbackSubmitted` 事件且参数一致

## 六、前端位置
- 房源详情页摘要：`apps/frontend/src/pages/ListingDetail.jsx`
- 独立房源评论页：`apps/frontend/src/pages/ListingReviewsPage.jsx`
- 租后评价：`apps/frontend/src/pages/ContractPage.jsx`
- 第一版暂不在 `MyContracts.jsx` 加“去评价”快捷按钮

## 七、gas 规则
- 第一版默认由租客自己支付租后评价的链上 gas。
- 不引入额外 gas 补贴或代付机制。
- 如果后续评价使用率太低，再讨论链下签名 + 平台代提交。

## 八、第一版明确不做
- 不做 ZK 评价
- 不做隐藏单条星级
- 不做评论密文
- 不做双向互评
- 不做房东评价租客
