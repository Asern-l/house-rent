# 租后评价与房源反馈方案

## 1. 目标
- 区分“房源反馈”和“租后评价”两类内容。
- 房源反馈公开展示，但不计入正式星级。
- 租后评价只允许真实租客提交，并参与正式平均分。
- 两类评论正文都走 IPFS，链上保存 `commentHash + commentCid` 锚点。
- 独立 verifier 能按 `commentCid` 联动读取正文并完成材料级校验。

## 2. 两类内容的边界
### 2.1 房源反馈
- 入口：
  - 房源详情页摘要入口
  - 独立评论页 `ListingReviewsPage`
- 对象：登录用户，且不是该房源房东。
- 作用：表达看房阶段或意向阶段反馈，例如“与描述不符、图片过旧、沟通一般”。
- 链上锚定：`feedbackType + commentHash + commentCid + listingId + sender + createdAt`
- 数据库存储：评论明文、反馈类型、钱包地址、评论 CID、交易哈希
- 不参与正式星级

### 2.2 租后评价
- 入口：合同详情页底部
- 对象：真实租客，且仅合同租客本人可提交
- 方向：单向，仅租客评价房源
- 链上锚定：`rating + commentHash + commentCid + contractId + listingId + tenant + ratedAt`
- 数据库存储：评论明文、星级、权重、评论 CID、交易哈希
- 参与正式平均分

## 3. 租后评价资格规则
- 一份合同最多评价一次。
- 仅合同租客本人可发起链上评价交易。
- 合同必须已正常结束。
- 评价窗口：合同结束后 30 天内。
- 续租不影响老合同评价；每份合同独立评价、独立计权。

## 4. 正式平均分规则
- 仅租后评价参与平均分。
- 权重按租期月数计算：`weight = leaseMonths`，最小为 1。
- 计算方式：
  - `weightedScoreSum += rating * weight`
  - `weightSum += weight`
  - `weightedAverage = weightedScoreSum / weightSum`
- 至少 3 条真实租客评价后才公开平均分。
- 不足 3 条前，评价列表可公开，但平均分显示“样本不足”。

## 5. 链上设计
### 5.1 合约函数
- `submitRentalReview(string contractId, bytes32 commentHash, uint8 rating, string commentCid)`
- `submitListingFeedback(string listingId, uint8 feedbackType, bytes32 commentHash, string commentCid)`

### 5.2 合约校验
- `commentHash != 0`
- `commentCid` 非空
- `rating` 在 `1..5`
- `msg.sender` 与链上角色匹配
- 评价时间窗口合法
- 合同存在且尚未重复评价

### 5.3 链上事件
- `RentalReviewSubmitted(contractId, listingId, tenant, rating, commentHash, commentCid, ratedAt)`
- `ListingFeedbackSubmitted(listingId, sender, feedbackType, commentHash, commentCid, createdAt)`

## 6. IPFS 材料与哈希语义
- 评论正文不直接上链。
- 评论正文先保存为 IPFS JSON 材料。
- `commentCid` 用于定位这份评论材料。
- `commentHash` 不对整份 JSON 做哈希，而是对其中的 `commentText` 做：
  - `keccak256(commentText)`

这意味着独立 verifier 在读取评论时，必须：
1. 用 `commentCid` 取回 IPFS JSON
2. 解析出 `commentText`
3. 对 `commentText` 重算 `commentHash`
4. 与链上事件中的 `commentHash` 对比

## 7. 后端落地
### 7.1 新表 `listing_feedbacks`
字段：
- `listing_id`
- `author_wallet`
- `feedback_type`
- `comment_text`
- `comment_hash`
- `comment_cid`
- `tx_hash`
- `chain_env`
- `created_at`

### 7.2 新表 `contract_reviews`
字段：
- `contract_id`
- `listing_id`
- `tenant_wallet`
- `rating`
- `weight`
- `comment_text`
- `comment_hash`
- `comment_cid`
- `tx_hash`
- `chain_env`
- `created_at`

### 7.3 路由
- `POST /listings/:id/feedbacks/prepare-onchain`
- `POST /listings/:id/feedbacks`
- `POST /contracts/:id/review/prepare-onchain`
- `POST /contracts/:id/review/onchain`

说明：
- `prepare-onchain` 负责先上传评论正文到 IPFS，返回 `commentCid`。
- 回写接口负责验证交易回执、方法参数、链上事件与 `commentCid / commentHash`。

## 8. 前端位置
- 房源详情页摘要：`apps/frontend/src/pages/ListingDetail.jsx`
- 独立房源评论页：`apps/frontend/src/pages/ListingReviewsPage.jsx`
- 租后评价：`apps/frontend/src/pages/ContractPage.jsx`

## 9. 独立 verifier 联动
独立 verifier 在按 `listingId` 验房源时，应同步：
- 查询 `ListingFeedbackSubmitted`
- 查询 `RentalReviewSubmitted`
- 读取每条事件中的 `commentCid`
- 从 IPFS 拉取评论正文
- 对 `commentText` 做 `keccak256(commentText)`
- 与链上 `commentHash` 比较
- 租后评价额外展示 `rating`

独立 verifier 的“房源详情”页可直接返回：
- 当前房源
- 图片
- 房源反馈
- 租后评价
- 可选全部历史版本

## 10. 当前不做
- 不做 ZK 评价
- 不做隐藏单条星级
- 不做评论密文
- 不做双向互评
- 不做链上平均分聚合

## 11. 当前方案结论
- 房源反馈与租后评价都公开展示正文，但正文主存于 IPFS。
- 链上用 `commentHash + commentCid` 锚定评论材料。
- 独立 verifier 可在脱离平台账号系统的情况下，直接按 `commentCid` 验评论正文。
