# 智能合约逻辑方案（权限、校验、可修改）（讨论稿）

## 1. 设计目标
- 关键状态迁移由合约强约束，防绕过网站。
- 权限、版本、状态合法性全部链上校验。

## 2. 角色与权限
- `landlordWallet`：房源创建/房源内容更新。
- `tenantWallet`：租客签署、支付动作发起。
- `landlordWallet`：房东签署、必要确认动作。
- `admin`（可选）：仅用于紧急暂停、系统治理，不介入日常签署真相。

## 3. 房源合约函数建议
- `createListing(listingId, landlordWallet, contentHash, schemaVersion)`
- `updateListingContentHash(listingId, newHash, newVersion, schemaVersion)`
- `setListingStatus(listingId, status)`

校验要点：
- `listingId` 不可重复创建。
- 仅房东地址可更新自身房源。
- `version` 必须递增。

## 4. 合同合约函数建议
- `createContract(contractId, listingId, tenantWallet, landlordWallet, contentHash, expiresAt, version)`
- `signTenant(contractId)`
- `signLandlord(contractId)`
- `recordInitialPayment(contractId, txRef)` 或支付函数内联
- `cancelDraft(contractId)`（仅双方未签）
- `markExpired(contractId)`（可按调用时检查到期）
- `renewContract(oldContractId, newContractId, newHash, newVersion, expiresAt)`

校验要点：
- 签署顺序与状态机严格匹配。
- 超时签约拒绝。
- 任一方签后禁止取消。

## 5. 事件设计（审计）
- `ListingCreated`
- `ListingContentHashUpdated`
- `ListingStatusChanged`
- `ContractCreated`
- `ContractSigned(contractId, role, signer, signedAt)`
- `ContractContentHashUpdated`（若同ID修订）
- `ContractRenewed`
- `ContractStatusChanged`

## 6. 版本与兼容策略
- 通过 `schemaVersion` 管理哈希规则升级。
- 旧版本可只读，新增流程走新版本函数。
- 服务器侧按版本重算对应哈希规则。

## 7. 风险控制建议
- 增加 `Pausable`（紧急停机）。
- 关键操作加 `nonReentrant`。
- 明确 gas 上限与批量操作边界。
- 上线前完成单元测试 + 状态机测试 + 回归脚本。
