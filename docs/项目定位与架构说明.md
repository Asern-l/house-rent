# 项目定位与架构说明

## 1. 项目定位
本项目不是“完全去中心化租房平台”，而是一个采用混合架构的链上租房原型：

- 产品层和治理层保持中心化
- 关键事实和关键状态保持可验证
- 交易准入由平台控制，验真材料尽量脱离平台单点

更准确的定位是：

- **中心化交易与治理层 + 链上可验证真相层 + IPFS 验真材料层**

这个定位不是妥协，而是基于当前业务特点做出的工程选择。

## 2. 为什么不追求完全去中心化
租房业务天然包含大量不适合完全去中心化处理的部分，例如：

- 账号与角色体系
- 风控与冷却期
- 默认展示与内容治理
- 搜索、筛选、分页、聚合
- 用户资料与隐私信息
- 评论与反馈管理

这些能力如果全部交给链上或纯内容网络处理，复杂度会显著上升，且用户体验会明显下降。

因此项目选择：

- 把产品治理和交易准入保留在平台层
- 把关键事实约束和可验真能力交给链上
- 把验真所需明文材料尽量外置到 IPFS

## 3. 核心设计原则

### 3.1 平台负责体验和治理
平台服务器负责：

- 用户注册、登录、角色识别
- 房源与合同的默认展示
- 风控拦截与冷却规则
- 搜索、筛选、聚合
- 交易前参数准备和错误提示
- 交易许可签发
- 链上交易回写校验
- 平台收录规则

也就是说，平台不是被动存储层，而是产品行为的组织者和交易门禁。

### 3.2 链上负责关键事实约束与监督
智能合约负责：

- 房源关键状态
- 合同关键状态
- 合同签名锚定
- 支付记录
- gas 锁仓与撤销
- 租后评价与房源反馈锚定

链上的作用不是替代整个平台，而是保证：

- 平台不是唯一真相源
- 平台不能随意篡改关键事实
- 用户可独立验证关键业务记录
- 没有平台许可的关键交易不会被放行

### 3.3 IPFS 负责验真材料，不负责业务判断
IPFS 层负责：

- 房源图片
- 房源公开快照
- 房源反馈明文
- 租后评价明文
- 历史版本快照
- 验真报告快照
- 合同全文密文与合同 PDF 密文

IPFS 的职责是“保存材料”，不是“判断业务状态”。

### 3.4 默认视图和链上事实分开
平台页面展示的“正式记录”和链上实际存在的记录不是同一个概念。

需要明确区分：

- **平台状态**
  - 是否已收录
  - 是否已展示
  - 是否已通过平台校验
  - 是否被隐藏或限流

- **链上状态**
  - 是否存在
  - 是否已确认
  - 哈希是否匹配
  - 事件是否匹配

这种分层可以避免“平台没展示 = 记录不存在”的误解。

## 4. 当前架构分层

### 4.1 前端
前端负责：

- 页面交互
- 钱包连接
- 请求服务器许可后再发链上交易
- 调用后端 API
- 展示链上与平台双重状态
- 拉取 IPFS 验真材料并做结果可视化

前端不是只会“发交易”，而是承担了交易前预检查、许可请求、网络切换、健康检查和错误翻译。

### 4.2 后端
后端负责：

- 账号体系
- 鉴权
- 风控
- 房源与合同业务编排
- 关键交易 permit / attestation 签发
- 链上交易回写校验
- IPFS 上传、CID 管理与材料索引
- 评价与反馈收录
- 默认视图和聚合数据输出

后端的定位不是简单数据库接口，而是业务接入层和治理层。

### 4.3 智能合约
合约负责：

- 存储关键链上状态
- 记录关键事件
- 验平台许可签名
- 拒绝不符合规则的关键写操作
- 为验真提供独立真相来源

合约是约束层，不是全功能应用层。

### 4.4 数据库
数据库负责：

- 平台默认状态
- 列表聚合
- 评价与反馈的索引与平台收录状态
- 用户资料
- 平台收录记录
- 链上回写台账
- IPFS `CID/hash` 索引

数据库不是唯一真相层，但仍然是产品层运行的主数据源。

## 5. 当前“去中心化”体现在哪里
虽然项目不是完全去中心化，但已经有明显的链上特征：

1. 合同不是只存在数据库里，而是有链上锚定
2. 首笔支付不是只信后台记录，而是有链上事件
3. gas 锁仓不是口头约定，而是链上约束
4. 租后评价和房源反馈都有链上哈希锚定
5. 验真页面可以比对链上记录与 IPFS 材料

因此，项目的区块链价值体现在：

- **可验证**
- **可追溯**
- **不可随意篡改**

而不是体现在“所有内容都链上化”。

## 6. 当前哪些内容仍应保持中心化
以下内容目前适合继续保留在中心化服务中：

- 账号登录与角色识别
- 风控规则
- 交易许可签发
- 默认列表展示
- 搜索筛选与聚合统计
- 收录规则与治理动作
- 平台治理动作
- 日志与观测

这些能力决定了：

- 产品是否好用
- 团队是否容易运维
- 风险是否可控

## 7. 当前哪些能力依赖链上
以下能力已经明显依赖链上真相：

- 房源关键状态存在性
- 合同关键事实
- 关键交易许可是否有效
- 支付确认
- gas 锁仓与退款逻辑
- 租后评价哈希
- 房源反馈哈希
- 验真依据

也就是说，链上已经不仅是展示点，而是业务闭环的一部分。

## 8. 服务器宕机时的真实含义
当前阶段，服务器不仅是业务主入口，还是关键交易的准入门禁。

因此：

- 服务器宕机时，不代表链上数据消失
- 但关键交易会不可用
- 验真在资料已上 IPFS 的前提下仍可继续

例如：

- 搜索与列表
- 我的房源 / 我的合同
- 默认评论页
- 风控判断
- 平台回写收录
- permit 签发

这说明当前架构仍然是：

- 平台为交易与治理入口
- 链上为真相锚点
- IPFS 为验真材料层

## 9. 未来可以增强的方向
如果后续继续迭代，可以沿下面方向增强，而不改变当前项目的基本定位：

### 9.1 内容外置
把适合外置的静态内容转成：

- 合同全文密文 / PDF 密文上传到加密 IPFS
- 房源公开资料、评价、反馈、快照上传到公开 IPFS
- 链上只锚定 `hash/CID`

适用对象：

- 图片
- 合同附件
- PDF
- 评论或反馈归档文本

### 9.2 第三方索引与独立验真
增加：

- 第三方 indexer
- 独立链上事实重建
- 更公开的治理日志

这样可以降低平台单点解释权。

### 9.3 治理日志增强
将以下动作至少做成可追踪日志：

- 房源隐藏
- 评论屏蔽
- 账号限制
- 风控拦截
- 收录拒绝

未来可以考虑：

- 日志哈希定期上链
- 多签治理

### 9.4 交易许可签名地址后续拆分
当前可以采用：

- 部署账号 = permit signer

但后续应支持：

- 将 `permitSigner` 配置成独立地址
- 只改 signer，不重做整套交易准入逻辑

这属于后续治理与权限隔离增强项。

### 9.5 更强的隐私方案
后续如果确有必要，再讨论：

- 加密内容分发
- 第三方内容授权
- ZK 匿名评价

这类能力属于增强项，不是当前主链路的前提。

### 9.6 身份绑定与第三方实名签署
身份绑定与第三方实名/电子签的实际落地思路，单独见：

- `docs/身份绑定与第三方实名签署说明.md`

这份说明用于回答两个问题：

- 为什么原型阶段没有接入完整 KYC 与第三方电子签
- 为什么未来若将身份摘要纳入合同材料，`content_json` 需要加密存储

## 10. 当前最准确的对外表述
建议项目对外统一使用以下口径：

### 简版
本项目采用“中心化交易与治理层 + 链上可验证真相层 + IPFS 验真材料层”的混合架构。平台负责交易准入、治理和默认展示，链上负责关键状态约束与关键事实锚定，IPFS 负责保存验真所需材料。

### 展开版
本项目不是完全去中心化平台，而是一个强调“平台可治理、关键事实可验证、验真材料可外置”的链上租房原型。账号体系、风控、交易许可、搜索、默认展示等依赖平台服务；合同哈希、签名锚定、支付记录、gas 锁仓、评价反馈哈希等关键事实依赖链上约束与验真；房源图片、公开快照、评论反馈明文、合同密文等验真材料则通过 IPFS 存储和索引。用户依赖平台完成正式交易与收录，也可以依据链上记录和 IPFS 材料独立核验关键结果。

## 11. 总结
当前项目最重要的价值，不在于“是否完全去中心化”，而在于：

- 平台仍然可治理
- 关键事实仍然可验证
- 验真材料不再必须依赖平台单点提供
- 产品体验和可信能力同时存在

这是一条现实、稳妥、可持续演进的路线。
