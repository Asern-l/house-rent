# 密码学增强与治理机制待办

## 文档目的
- 记录当前项目在 IPFS 之外，后续仍值得补齐的密码学增强机制。
- 只记录“有明确业务价值、和当前架构兼容”的方向，不讨论过重或偏离主线的方案。

## 一、当前推荐优先级
建议优先级从高到低如下：

1. 交易准入 permit 签名
2. 合同全文加密
3. 治理日志哈希锚定
4. 多签管理员
5. Merkle 批量锚定
6. commit-reveal（仅在确有延迟公开需求时）

## 二、交易准入 permit 签名

### 目标
- 强制关键交易先经过平台服务器。
- 合约只接受带平台签名许可的交易。

### 用途
适用动作：
- 房源发布
- 合同创建
- 租客签署
- 房东签署上链
- 首笔支付
- 租后评价
- 房源反馈

### 最低设计要求
permit 至少应包含：
- `action`
- `sender`
- `targetId`
- `deadline`
- `nonce`
- 可选参数摘要 `paramsHash`

### 合约侧要求
- 验平台 signer 签名
- 校验 `deadline`
- 校验 `nonce`
- 校验 `msg.sender`
- 校验动作与目标对象匹配

### 价值
- 平台仍保留交易治理权
- 用户仍自己发交易、自己付 gas
- 不经过服务器的关键交易无法直接上链

## 三、合同全文加密

### 目标
- 合同全文和合同 PDF 可以外置存储，但不能明文公开。

### 当前建议
- 明文合同不直接公开上 IPFS
- 合同正文和 PDF 先加密，再上传 IPFS

### 第一版建议
- 使用对称加密
- 推荐：
  - `AES-256-GCM`
  - `Argon2id` 或 `scrypt` 派生密钥
- 每份合同独立：
  - `salt`
  - `iv`
  - `tag`

### 后续增强
- 当前可先允许“部署账号 = permit signer”
- 未来可进一步演进为：
  - 随机合同密钥
  - 用双方钱包分别加密该合同密钥

### 价值
- 内容可外置
- 验真材料可保留
- 合同明文不被被动公开

## 四、治理日志哈希锚定

### 目标
- 平台治理动作可追踪、可证明、不可事后无痕篡改。

### 适用范围
- 房源隐藏
- 评论屏蔽
- 账号限制
- 风控拦截
- 收录拒绝

### 第一版建议
- 治理明文日志继续保留在服务器
- 按批次计算：
  - 总 hash
  - 或 Merkle root
- 定期上链锚定

### 价值
- 平台仍有治理权
- 但治理动作不是完全黑箱
- 后续答辩和对外展示时更容易说明“平台可治理，但不是不可监督”

## 五、多签管理员

### 目标
- 降低关键治理权限集中在单一私钥上的风险。

### 适用对象
- permit signer 更新
- 关键参数更新
- 管理员权限变更
- 治理日志锚定提交者

### 当前建议
- 第一版可以不立刻上多签
- 但合约权限设计要为后续多签治理留空间

### 价值
- 平台治理权更可信
- 风险更分散
- 更适合后续对外讲“关键权限不是单人可任意改动”

## 六、Merkle 批量锚定

### 目标
- 降低逐条上链成本，同时保留批量材料可验证能力。

### 适用对象
- 评论归档
- 反馈归档
- 历史版本快照
- 验真报告快照
- 治理日志批次

### 第一版建议
- 不把每条材料都单独上链
- 按批次生成：
  - Merkle tree
  - root 上链

### 价值
- gas 成本更低
- 保留批量材料的可验性
- 比“全量逐条上链”更现实

## 七、commit-reveal（可选）

### 目标
- 支持“先承诺、后公开”的业务场景。

### 适用场景
- 延迟公开评价
- 密封报价
- 阶段性不可提前泄露的信息提交

### 当前判断
- 如果评价体系最终公开星级和评论，则不是当前优先项
- 仅在后续确实需要“先提交承诺、后揭示内容”时再做

## 八、当前不建议投入的方向

### 1. ZK 匿名评价
原因：
- 复杂度太高
- 与当前主链路收益不成正比
- 适合后续研究型增强，不适合当前主线优先项

### 2. 全量链上密文评论系统
原因：
- 成本高
- 交互复杂
- 当前没有明确必要

### 3. 去中心化身份大系统
原因：
- 会把项目重心从租房主流程拉偏
- 当前阶段收益不高

## 九、建议落地顺序
1. 先做 permit 机制。
2. 再做合同全文加密与 IPFS 外置。
3. 再补治理日志哈希锚定。
4. 后续视需要再做多签管理员。
5. 最后再考虑 Merkle 批量锚定和 commit-reveal。
