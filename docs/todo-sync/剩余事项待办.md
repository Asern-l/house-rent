# 剩余事项待办

## 文档目的
- 只保留当前路线下仍值得继续推进的边角事项。
- 不再重复已实现主链路，也不保留已经放弃的方向。

## 一、交易准入 permit 剩余事项

### 1. signer 与部署账号拆分
- 当前允许：
  - `deploy admin = permit signer`
- 后续应支持：
  - `deploy admin`
  - `permit signer`
  使用不同地址。

### 2. signer 更新入口
- 合约当前通过构造函数注入 `trustedSigner`。
- 后续如需切换 signer，应增加：
  - 管理员更新入口
  - signer 更新事件
- 目标是切换 signer 时不必重做整套交易准入逻辑。

### 3. permit 可观测性
- 验真页或管理页后续可补：
  - 本次交易是否带 permit
  - permit signer 是谁
  - 是否与当前平台 signer 一致

## 二、治理与权限收口

### 4. 治理日志哈希锚定
- 平台治理动作当前主要留在服务器日志。
- 后续可考虑按批次做：
  - 总 hash
  - 或 Merkle root
  上链锚定。

### 5. 多签管理员
- 当前不急着做。
- 但如果后续要把 signer 更新、关键参数更新做得更稳，建议补：
  - 单独管理员角色
  - 或多签管理员

## 三、链上状态清洁能力

### 6. 已到期合同的显式释放入口
- 当前现状：
  - 合同当前是否仍有效，验真与展示都按时间语义优先判断
  - `Completed` 只保留为链上终态收口，不作为过期主判断前提
  - 当前链上主路径依赖懒释放：
    - `createContractRecord(...)`
    - `recordInitialRentPayment(...)`
  - 合约已提供 `completeExpiredContract(contractId)`，但没有正式业务入口
- 当前核心机制已具备：
  - 懒释放
  - `completeExpiredContract(...)`
- 但仍没有正式业务入口。
- 若后续需要更干净的链上状态，可再补：
  - 前端按钮
  - 预检查接口
  - 独立日志闭环

### 7. 显式释放的展示条件
- 仅在以下条件同时满足时才考虑展示：
  - 合同按时间已过期
  - 链上仍保留旧占用语义
  - 当前房源仍指向该链头

### 8. 保持当前边界
- 不把显式释放当成主业务步骤。
- 不把 `Completed` 当成过期判断前提。
- 继续保持“时间优先判断有效性，链上终态负责收口”的口径。
