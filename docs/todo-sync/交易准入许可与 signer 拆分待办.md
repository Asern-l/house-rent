# 交易准入许可与 signer 拆分待办

## 文档目的
- 记录“交易强制经过服务器、验真材料走 IPFS”这条路线下仍待补齐的事项。
- 记录 permit signer 当前可与部署账号合并使用，但后续需要支持拆分的设计约束。

## 一、当前建议基线
- 关键交易采用服务器许可签发模式。
- 前端先请求服务器，再由服务器做：
  - 权限校验
  - 风控
  - 状态检查
  - 参数准备
  - permit 签名
- 用户再使用自己的钱包提交链上交易。
- 合约侧校验：
  - 签名者地址
  - deadline
  - nonce
  - sender
  - action
  - 关键参数匹配

## 二、当前可接受的第一版
- 部署账号与 permit signer 可以先使用同一个地址。
- 目的：
  - 降低第一版复杂度
  - 先把交易准入机制打通

## 三、后续必须保留的拆分能力
后续应支持把：

- `deploy admin`
- `permit signer`

拆成不同地址。

要求：
1. 不重做整套交易准入逻辑。
2. 只通过更新 `permitSigner` 即可切换签名者。
3. 不要求重新部署全部业务流程。

## 四、待补齐的设计点

### 1. 合约侧
- 增加 `permitSigner` 存储变量。
- 增加只允许管理员更新 signer 的入口。
- 明确 signer 更新事件。
- 明确 nonce 防重放机制。

### 2. permit 结构
建议至少包含：
- `action`
- `sender`
- `targetId`
- `deadline`
- `nonce`
- 可选参数摘要

### 3. 管理权限
- 明确谁能更新 signer。
- 明确是否采用单管理员还是后续多签。

### 4. 文档与验真
- 后续验真页应展示：
  - 本次交易是否带 permit
  - permit signer 是否匹配当前平台 signer

## 五、推荐顺序
1. 先落 permit 机制本身。
2. 再把 signer 地址从硬编码/单地址思路收成可配置变量。
3. 最后再做 signer 和部署账号的权限拆分。
