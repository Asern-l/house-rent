# IPFS 与本地验真服务待办

## 文档目的
- 记录“交易强制经过平台服务器、验真材料走 IPFS”路线下，仍待补齐的独立验真能力。
- 明确后续应支持一个不依赖平台业务数据库的本地验真服务。

## 一、当前已定方向
- 交易主流程仍由平台服务器控制与收录。
- 关键状态与关键事实由链上约束和锚定。
- 验真所需材料逐步迁移到 IPFS：
  - 房源图片
  - 房源公开快照
  - 房源反馈明文
  - 租后评价明文
  - 合同全文密文
  - 合同 PDF 密文

## 二、本地验真服务待办
后续应增加一个“本地验真服务”或等价本地验真工具，其目标是：

1. 不依赖平台业务数据库  
2. 不依赖平台默认展示接口  
3. 只依赖：
   - 合约地址
   - ABI
   - 链上事件与状态
   - IPFS 材料

### 本地验真服务至少应支持
- 按 `listingId` 验证房源快照与链上哈希是否一致
- 按 `contractId` 验证合同内容与链上哈希是否一致
- 验证支付交易、评价交易、反馈交易是否真实存在且事件匹配
- 拉取 IPFS 内容并本地重算：
  - `contentHash`
  - `commentHash`
  - `feedbackHash`
- 输出可读的验真结果，而不是只返回原始 RPC 数据

### 本地验真服务的意义
- 防止平台成为唯一解释者
- 允许第三方或答辩现场离开平台主服务独立核验
- 为后续“服务器可用但不是唯一真相源”提供真正落地的能力

## 三、建议形态
第一版可以是以下任一形式：

1. 本地 Node.js 验真服务
2. 独立 CLI 验真脚本
3. 前端验真页的“本地模式”

建议优先级：
- 先 CLI / 本地脚本
- 再独立本地服务

## 四、推荐顺序
1. 先把验真材料的 `CID/hash` 结构补齐。
2. 再把房源、合同、评价、反馈的 IPFS 材料读取逻辑补齐。
3. 然后实现本地验真脚本或本地验真服务。
4. 最后再考虑第三方 indexer 或更强的离线验真页面。
