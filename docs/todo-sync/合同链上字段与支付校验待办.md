# 合同链上字段与支付校验待办

## 文档目的
- 记录合同链上模型后续仍待补齐的事项。
- 记录已确认但尚未完全落地的签名、续约、提前解约与链上校验规则。
- 本文档只写剩余待办，不重复列出已实现能力。

---

## 一、当前已定方案（作为后续实现基线）
- 合同链上主字段已采用：`contractId`、`listingId`、`parentContractId`、`renewalChildContractId`、`tenant`、`landlord`、`contentHash`、`initialAmountWei`、`startAtMs`、`endAtMs`、`createdAt`、`tenantMessageHash`、`landlordMessageHash`、`tenantSignedAt`、`landlordSignedAt`、`status`、`exists`。
- 合同支付窗口采用合约级全局参数：`paymentWindowMs`。
- 部署脚本通过 `PAYMENT_WINDOW_HOURS` 传入 `paymentWindowMs`。
- `activeContractByListing` 当前语义已调整为“合同链头索引”，不再表示“当前生效合同”。
- 续约合同仅允许租客在父合同页发起。
- 一个父合同同一时间只允许一个子合同。
- 续约条款优先继承父合同最终条款；父合同缺失时回退房东房源默认条款。
- 签名锚点方案为：`messageHash` 存 storage，完整 `signature` 发 event，完整消息与签名结果放入 PDF/数据库。

---

## 二、支付与生效规则待办

### 1. 明确 `Active` 与“当前有效占用”的区别并贯穿所有查询
- 重点原则：`status == Active` 只表示“这份合同曾经被激活/支付成功过”。
- 它不直接等于“当前仍有效占用”。
- 当前有效占用仍需结合：
  - `startAtMs`
  - `endAtMs`
  - 父子续约关系
- 待办：前后端所有房源状态展示、验真、占用判断接口都要统一使用这套语义。

### 2. 续约提前支付后的展示态与查询态统一
- 当前续约支持提前支付。
- 当父合同尚未结束时，子合同支付成功后不能直接按“当前生效合同”处理。
- 前端展示应统一使用“已支付待接续”或等价文案。
- 待办：
  - 合同列表页
  - 房源详情页
  - 我的合同页
  - 链上验真页
  均需统一这套文案和判定逻辑。

### 3. 后端数据库状态与链上状态的进一步收口
- 当前数据库仍使用：`pending`、`tenant_signed`、`pending_payment`、`active`、`ended`、`cancelled`、`expired`。
- 链上状态机与数据库状态机仍不是完全一一对应。
- 待办：明确数据库 `active` 在续约提前支付场景下是否仍保留，还是后续引入更明确的展示层派生状态。

## 三、续约合同待办

### 4. 父合同唯一子合同约束的链下配套
- 链上已按“一个父合同同一时间只允许一个子合同”建模。
- 待办：后端和前端继续补齐显式提示，避免用户在 UI 上反复尝试创建第二个子合同。

### 5. 续约取消后的父合同回收逻辑
- 规划规则：若子合同取消，则父合同的 `renewalChildContractId` 应清空。
- 当前取消功能尚未完整链上化。
- 待办：在取消专题实现时，一并补齐父合同对子合同引用的清理逻辑。

### 6. 续约链的“当前有效合同”统一查询入口
- 当前模型要求：从链头合同顺着 `renewalChildContractId` 向下判断当前有效合同。
- 待办：为前端和后端统一提供稳定入口，避免各处各写一套遍历逻辑。
- 形式可二选一：
  1. 合约提供更明确的 getter
  2. 后端封装统一链式查询工具
