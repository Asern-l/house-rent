# 房源上链与合同上链目的说明

## 文档目的
- 说明“房源上链”和“合同上链”分别解决什么问题。
- 明确两者在交易追溯、纠纷处理、用户决策中的作用边界。
- 标记当前已实现与待实现能力，避免误解。

## 1. 合同上链的目的

### 1.1 核心目标
- 固化“某份合同内容在某个时间点真实存在且未被篡改”。
- 通过合同哈希（`contractHash`）提供可验证的证据锚点。
- 让后续支付记录、纠纷记录都可以围绕 `contractId` 聚合查询。

### 1.2 对业务的价值
- 防篡改：合同正文如被改动，重算哈希后会与链上哈希不一致。
- 可追溯：支付、取消、争议都能引用同一个 `contractId`，形成同一证据链。
- 可核验：第三方可基于合同文件与链上记录独立验证，不依赖平台单点数据库。

### 1.3 你关心的“绑定关系”
- 合同上链后，后续交易与纠纷应绑定该合同：
  - 交易：支付记录引用 `contractId`（当前已实现首笔一次性支付绑定）。
  - 纠纷：争议工单/仲裁记录引用 `contractId`（待实现）。

## 2. 房源上链的目的

### 2.1 核心目标
- 固化房源“存在性与归属”的基础事实（如房源ID、房东地址、创建时间）。
- 让房源成为可被引用的链上对象，承接该房源下的合同与评价索引。
- 当前说明：房源上链改为“手动触发”模式，不再在发布房源时自动上链。

### 2.2 对业务的价值
- 房源可信底座：证明某房源在某时由某地址发布，降低“凭空造房源”争议。
- 关联视图：可按房源维度查看其合同履约情况（通过 `listingId -> contractId` 关系）。
- 决策参考：未来可叠加“引用该房源的评价上链”形成公开信用画像（待实现）。

### 2.3 你关心的“买家参考”
- 买家可参考两类信息：
  - 该房源关联合同的履约情况（是否生效、是否发生支付、是否存在纠纷）。
  - 引用该房源的评价记录（评价上链待实现）。

## 3. 房源与合同的联动关系
1. 房源上链：建立房源锚点（`listingId`）。
2. 合同上链：合同记录引用 `listingId`，并固化 `contractHash`。
3. 交易/纠纷：后续记录引用 `contractId`，再通过合同反查到 `listingId`。
4. 评价（待实现）：评价记录直接引用 `listingId`，形成房源长期信誉层。

## 4. 当前实现状态（截至当前版本）
- 已实现：
  - 合同哈希上链与验真。
  - 首笔一次性支付记录与合同绑定（`contractId`）。
- 已部分实现（合约侧能力已接入手动触发）：
  - 支持房东手动触发房源上链重试（非自动发布即上链）。
- 未实现：
  - 房源上链自动闭环（发布即上链、自动失败补偿、验真联动）。
- 待实现：
  - 评价上链并引用房源（`listingId`）。
  - 纠纷/仲裁记录上链并引用合同（`contractId`）。
  - 房源维度的一站式链上履约画像查询接口。

## 5. 设计结论
- 合同上链偏“证据与执行追溯”。
- 房源上链偏“对象锚点与信誉聚合”。
- 两者结合后，才能同时满足：
  - 合同内容可信（防篡改）
  - 交易与纠纷可追踪（绑定合同）
  - 房源历史可评估（绑定合同与评价）
