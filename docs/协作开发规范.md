# 协作开发规范（house-rent）

## 1. 项目基线
- 项目名称：`house-rent`（CCL Housing / 信源链租房原型）
- 项目路径：`D:\0Project\house-rent`
- 技术栈：
  - 前端：React + Vite + Tailwind + ethers
  - 后端：Node.js + Express + sql.js
  - 合约：Solidity + Hardhat
- 项目目标：持续迭代“房源发布 -> 合同签署 -> 合同哈希上链 -> 验真查询”主流程，并逐步闭环预留功能。

## 2. 目录与模块约定（以当前仓库为准）
- `apps/backend`：后端核心逻辑（鉴权、房源、合同、验真、数据层）
- `apps/frontend`：前端页面与业务交互（登录、房源、合同、验真）
- `blockchain`：智能合约、Hardhat 配置、部署脚本
- `scripts`：启动脚本、回归脚本、辅助脚本
- `docs`：项目文档（模块说明、接口文档、日志说明、待实现清单）
- `logs`：运行日志输出目录

说明：不强制重命名为 `/backend`、`/frontend`、`/utils`，统一按现有目录维护。

## 3. 代码规范
- 每个新增或修改的函数需保留中文函数编号注释，格式示例：

```js
// 函数 1: 功能说明
function example() {
  // ...
}
```

- 函数注释与函数实现必须在同一文件中。
- 核心功能代码必须可运行；新增逻辑需提供最小可验证路径（页面操作、接口调用或脚本回归）。
- 冗余代码应清理；复杂功能可拆分文件，但要保持模块边界清晰。

## 4. 日志规范
- 后端日志统一复用 `apps/backend/src/logger.js`（或其后续统一日志入口）。
- 核心函数的输入、输出、关键判断、异常分支应记录日志。
- 日志点建议与函数编号对应，便于排查。
- 日志测试点文档维护在：`docs/日志埋点说明.md`。

## 5. 文档更新规范
- 代码变更后，必须同步更新 `docs`，至少覆盖以下文档：
  - 模块与文件职责：`docs/项目结构与文件职责说明.md`
  - 接口说明：`docs/后端接口与前端调用文档.md`
  - 日志测试点：`docs/日志埋点说明.md`
- 新增功能如影响现状结论，应同步更新：
  - `docs/已实现模块说明.md`
  - `docs/预留字段与未闭环功能清单.md`
- 新增文件/文件夹时，必须在“项目结构与文件职责说明”中登记用途。

## 6. 接口文档模板（沿用）
新增接口时，在 `docs/后端接口与前端调用文档.md` 中补齐以下字段：
- 接口名称
- 请求路径与方法
- 鉴权要求
- 请求参数（必填/选填、类型、含义）
- 返回值（成功/失败结构）
- 示例调用（请求与响应示例）

## 7. 执行与验收约束
- 所有改动优先落盘到仓库文件，不以聊天文本替代代码/文档提交。
- 变更完成后至少执行一项验证（如前端构建、后端启动、签署流程脚本）。
- 若存在环境依赖（例如 `.env`），优先维护对应 `env.example`，确保可复现。
