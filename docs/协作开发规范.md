# 协作开发规范（house-rent）

## 1. 项目基线
- 项目名称：`house-rent`
- 项目路径：`D:\0Project\house-rent`
- 技术栈：
  - 前端：React + Vite + Tailwind + ethers
  - 后端：Node.js + Express + sql.js
  - 合约开发：Solidity
  - 合约工程：Hardhat（编译/部署）
  - 本地链节点：Ganache（持久化本地网络）
  - 链交互库：ethers（前后端）
- 目标：持续迭代“房源发布 -> 合同签署 -> 合同哈希上链 -> 验真查询”主流程，并逐步闭环预留功能。

## 2. 目录与模块约定
- `apps/backend`：后端核心逻辑（鉴权、房源、合同、验真、数据层）
- `apps/frontend`：前端页面与业务交互
- `blockchain`：智能合约、Hardhat 配置、部署脚本
- `scripts`：启动、重置、回归脚本
- `docs`：项目文档
- `logs`：运行日志输出目录
- `apps/backend/data/uploads`：运行时用户上传目录（仅本地/服务器数据，不入 Git）

## 3. 代码规范
- 新增或修改函数需保留中文函数编号注释。
- 函数注释与函数实现必须在同一文件。
- 核心功能必须可运行并可验证。
- 冗余代码应清理，复杂逻辑可拆分文件。

## 4. 日志规范
- 后端日志统一复用 `apps/backend/src/logger.js`（或后续统一日志入口）。
- 核心函数输入、输出、关键判断、异常分支应记录日志。
- 必须覆盖两类错误日志：
  - `api-error`：包含 404、中间件 500、以及请求完成时的 4xx/5xx。
  - `system-error`：包含 `unhandledRejection`、`uncaughtException`。
- 新增或修改接口时，需确认“业务校验失败（4xx）”与“异常失败（5xx）”都可追踪到日志。
- 日志测试点维护在 `docs/日志埋点说明.md`。

## 5. 文档规范（强制）
- 代码变更后必须同步更新 `docs`。
- 涉及接口、错误处理、日志策略变更时，必须同步更新：
  - `docs/日志埋点说明.md`
  - `docs/后端接口与前端调用文档.md`（若接口行为变化）
  - `docs/已实现总览.md` / `docs/todo-sync/*`（状态变化）
- 文档引用路径统一使用**相对路径**，禁止写绝对路径（如 `D:\...`）。
- 推荐示例：
  - `apps/backend/src/index.js`
  - `apps/frontend/src/app/App.jsx`
  - `docs/todo-sync/`
- 至少同步维护：
  - `docs/项目结构与文件职责说明.md`
  - `docs/后端接口与前端调用文档.md`
  - `docs/日志埋点说明.md`
  - `docs/已实现总览.md`
  - `docs/todo-sync/`

## 6. 接口文档模板（沿用）
新增接口时，在 `docs/后端接口与前端调用文档.md` 补齐：
- 接口名称
- 请求路径与方法
- 鉴权要求
- 请求参数（必填/选填、类型、含义）
- 返回结构（成功/失败）
- 示例调用（请求与响应）

## 7. 执行与验收
- 变更必须落盘到仓库文件，不以聊天文本代替提交。
- 变更完成后至少执行一项验证（后端启动、前端构建、回归脚本等）。
- 环境变量变更需同步维护对应 `*.env.example`。
- 运行时数据目录（如 `apps/backend/data/uploads/`、`logs/`、`apps/backend/data/`）不得提交到仓库。
