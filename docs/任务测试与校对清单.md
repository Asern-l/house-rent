# 任务测试与校对清单

## 1. 测试与校对主清单

### 1.1 基线准备
1. 安装依赖并确认版本一致：`apps/backend`、`apps/frontend`、`blockchain`。
2. 清理端口占用：`3000/3001/3002/3005/8545`。
3. 校验 `.env` 与前端网络地址配置（Sepolia / Local）。

### 1.2 启动与健康检查
1. 分别验证：
   - `scripts/start-sepolia-services.ps1`
   - `scripts/start-local-services.ps1`
   - `scripts/start-parallel-services.ps1`
2. 健康检查：
   - `GET /api/health`
   - `GET /api-local/health`
3. 前端网络切换时，选中网络不可用应有明显提示（红色状态）。

### 1.3 认证与账号
1. 注册、登录、重置密码全流程。
2. 共享账号库验证：同账号可在 sepolia/local 登录。
3. 钱包绑定策略验证：首次绑定、重连同地址、禁止换绑。

### 1.4 房源功能
1. 发布房源：无图/单图/多图。
2. 图片上传限制：格式、大小、数量、异常提示。
3. 我的房源：改价、上下架、删除图片、重新评分。
4. 房源上链状态：成功/失败/重试链路。

### 1.5 合同功能
1. 申请签约到合同创建与房源锁定。
2. 协商提案：提交、接受、定稿。
3. 双方签署状态机与错误码验证。
4. 支付授权：过期、重复、金额不符等拦截。
5. PDF 导出可用性与字段完整性。
6. 取消合同、提前解约、房源释放一致性。

### 1.6 定时任务与补偿
1. 超时未支付自动取消合同。
2. 到期合同自动结束并释放房源。
3. 房源/合同上链失败自动重试。
4. `GET /api/console/status` 与实际运行状态一致。

### 1.7 日志与错误覆盖
1. 覆盖 4xx/5xx/404/异常中间件日志。
2. 核对 sepolia/local 分流日志文件写入是否正确。
3. 关键链路日志字段齐全（requestId、userId、contractId/listingId 等）。

### 1.8 文档一致性校对
1. `docs/已实现总览.md` 每条已实现均有代码证据。
2. `docs/待实现总览.md` 保留真实未完成项，删除已落地项。
3. 脚本、开关、接口改动同步到对应文档。

## 2. old-docs 原文搬运：原待实现但现被宣称已实现的重点验证项

说明：
- 下列内容从 `old-docs/待实现总览.md` 原文搬运，用于“重点回归核对”。
- 不做语义改写，保持原文表达。

### 1. 钱包签名原文验签（P0）
- 现状：主要依赖账号权限和状态机，未对签署声明做标准化验签。
- 建议：前端签署前生成签名消息（合同ID、合同哈希、时间戳），后端验签并留存。

### 2. 签署证据字段补全（P1）
- 现状：证据以日志为主，结构化证据不足。
- 建议：补充签署IP、设备摘要、签署声明摘要、签署时间戳、请求ID等。

### 3. 错误码规范化（P1）
- 现状：错误信息以文本为主。
- 建议：定义统一错误码（如 `SIGN_STATUS_INVALID`、`WALLET_MISMATCH`）。

### 4. 合同 PDF 生成（P1）
- 现状：合同内容以 JSON 展示为主。
- 建议：关键节点生成 PDF（文本、签署时间、哈希、交易信息）。

### 5. MetaMask 双签名流程说明固化（P1）
- 现状：用户容易误解“签约一次签名即可完成全部流程”。
- 建议：在产品与技术文档固定说明“签约签名 + 支付签名”两步流程。

### 6. 支付后端授权签名 + 合约验签（P0）
- 现状：未实现一次性授权签名与链上验签准入。
- 建议：引入 `contractId/payer/amount/deadline/nonce` 授权模型，合约验签后放行支付。

### 7. 逾期不可支付的合约原生校验（P0）
- 现状：逾期拦截主要在后端状态机，存在绕过前端直连合约窗口。
- 建议：合约增加支付截止时间与状态校验，支付函数强制校验 `status` 与 `deadline`。

### 8. 上链失败补偿机制（P0）
- 现状：上链失败仍以人工重试为主。
- 建议：增加中间状态（如 `active_pending_onchain`）与后台重试任务。

### 9. 预付款能力（P1）
- 现状：已移除预付款实现，当前仅一次性支付。
- 建议：若恢复，需重建金额计算、状态流转、退款对账与验真链路。

### 10. 续费/续租支付能力（P1）
- 现状：已移除 `renewal`，仅保留 `initial`。
- 建议：按“续租提案 -> 双方确认 -> 新周期支付 -> 回写”独立实现。

### 11. 租金支付存证后端/前端闭环（P1）
- 现状：合约有 `recordRentPayment`，系统侧仍缺录入/查询完整体验优化。
- 建议：完善权限、查询视图、异常恢复与审计字段。

### 12. 租客与房东共同编辑合同内容（P0）
- 现状：合同正文主要由系统生成，协商补充机制不足。
- 建议：`提案 -> 修改 -> 定稿 -> 签署` 流程，配合版本与差异留痕。

### 13. 合同版本模型（P1）
- 现状：签署后缺少正式修订机制。
- 建议：增加 `version`、`parent_contract_id`，修订生成新版本并重签。

### 14. 提前解约模块（P1）
- 现状：缺少提前解约申请、协商、结算流程。
- 建议：支持双方发起/同意提前解约并固化结算结果。

### 15. 房源图片上传闭环（P1）
- 现状：已有 `image_urls` / `image_hashes` 字段与前端入参。
- 缺失：真实上传接口、图片管理流程、上传鉴权与防滥用。

### 16. 房源链上回写闭环（P1）
- 现状：已有 `listings.tx_hash` 与验真读取能力。
- 缺失：发布后上链、回写、失败重试与状态展示。

### 17. 房源管理增强：改价流程（P1）
- 现状：缺少完整改价流程与审计约束。
- 建议：改价接口、权限校验、操作日志，与签约中状态联动限制。

### 18. 反复申请签约占用房源治理（P0）
- 现状：可重复申请导致房源被频繁 `locked`。
- 建议：同租客同房源限频、全局申请频率限制、异常占位自动释放、风控日志。

### 19. 租客恶意签署不付款治理（P0）
- 现状：虽有超时取消，但对“签后不付”缺少用户维度约束。
- 建议：签后未付款计数、冷却机制、履约率风控视图。

### 21. 房源操作日志增强（P1）
- 现状：已有基础日志，结构化审计维度仍可增强。
- 建议：补充操作者、前后状态、请求ID、备注原因、来源端等字段。

### 22. 接入真实模型推理（P1）
- 现状：已移除随机评分逻辑。
- 建议：上传文本与图片后调用独立 AI 服务评分，输出分值、置信度、风险标签。

### 23. 评分可解释性（P1）
- 建议：保存评分原因（图片完整性、文本一致性、异常特征等）。

### 24. 图片哈希链上存证恢复（P1）
- 建议：在上传与哈希生成完成后恢复 `imageHashes` 上链。
- 说明：建议“可选上链”或“异步上链重试”，避免阻塞发布。

### 25. 评分版本管理（P1）
- 建议：记录 `model_version`、`scored_at`、`score_source`。

### 32. 测试网/本地链一体化控制台（P1）
- 现状：已实现前后端与数据库隔离切换，但缺少“一键全链路切换控制台”。
- 建议：统一管理前端网络、后端环境、脚本执行与状态检查。

### 33. 跨环境隔离自动化测试（P1）
- 现状：缺少 CI 级自动化隔离校验。
- 建议：增加 Sepolia/Local 双环境验证用例，防止串数据回归。

### 34. 本地链调试态数据伪造工具（P1）
- 目标：快速构造“运行数月后”的房源与合同状态用于演示与联调。
- 缺失：调试 RPC 封装、链下数据库联动脚本、非 local 环境防护。

### 35. 合约与前端 ABI 同步防漂移机制（P1）
- 现状：存在“合约变更后前端 ABI 可能滞后”的风险。
- 建议：把 ABI 同步、部署产物校验、前端地址回填纳入标准脚本与 CI 检查。
