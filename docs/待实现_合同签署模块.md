# 合同签署模块待实现清单

## 文档目的
- 记录当前“合同签署”模块尚未完善但具有必要性的事项。
- 作为后续迭代的实现参考，不影响当前可运行版本。
- 当前版本声明：押金功能暂未启用，本文涉及押金相关内容仅作为后续规划。

## 优先级说明
- P0：高必要性，建议优先实现。
- P1：中必要性，建议第二阶段实现。
- P2：增强项，可按时间选择实现。

## 待实现事项

### 1. 增加钱包签名原文验签（P0）
- 现状：当前主要依赖账号权限和状态机，未对签署声明进行标准化验签。
- 建议：前端签署前生成签名消息（含合同ID、合同哈希、时间戳），调用钱包签名；后端验签并保存签名结果。
- 价值：显著提升“是谁在何时同意了什么内容”的证据强度。

### 2. 上链失败状态细分与补偿机制（P0）
- 现状：合同可能“已生效但上链失败”，目前依赖人工重试。
- 建议：增加中间状态（如 `active_pending_onchain`），并提供后台重试或任务队列。
- 价值：避免业务状态与链上状态不一致，提升可恢复性。

### 3. 合同版本模型（P1）
- 现状：签署后缺少修订机制。
- 建议：新增 `version`、`parent_contract_id` 字段，修订生成新版本并重新签署。
- 价值：支持真实协商场景，避免直接覆盖原合同破坏证据链。

### 4. 签署证据字段补全（P1）
- 现状：签署证据以操作日志为主，结构化证据不足。
- 建议：补充签署IP、设备信息、签署声明摘要、签署时间戳、请求ID等字段并固定格式存储。
- 价值：争议时可提供更完整的取证材料。

### 5. 错误码规范化（P1）
- 现状：错误信息以文本为主，机器可识别度不高。
- 建议：定义统一错误码（如 `SIGN_STATUS_INVALID`、`WALLET_MISMATCH`、`ONCHAIN_RETRY_REQUIRED`）。
- 价值：前端可精准提示，日志统计更稳定。

### 6. 实名/CA 体系预留接口（P2）
- 现状：仅钱包地址绑定，未引入实名或证书能力。
- 建议：预留实名认证状态字段和外部回调接口，不立即接第三方。
- 价值：为后续合规升级预留架构扩展点。

### 7. 房源状态与签约并发保护（P0）
- 现状：房源上下架与合同创建/签署可能并发发生，存在状态竞态风险。
- 建议：在创建合同、房源上下架时增加事务级状态校验与原子更新（如条件更新、乐观锁版本号或行锁策略）。
- 价值：避免“同一时刻既下架又签约”导致的数据不一致问题。

### 8. 房源操作日志（P1）
- 现状：房源上架/下架/状态变更缺少结构化审计日志。
- 建议：记录操作者、操作类型、目标房源、变更前后状态、时间、请求ID、备注原因。
- 价值：便于问题追溯、运营审计与风控分析。

### 9. 房源审核流（P2）
- 现状：房源关键变更操作为即时生效，缺少审核关口。
- 建议：为高风险操作增加“待审核 -> 已通过/已拒绝”流程，并保留审核意见与审核人。
- 价值：降低误操作与恶意操作风险，支持平台治理。

### 10. 房源上链后置实现（P2，最终阶段）
- 现状：房源上链会增加钱包交互、链上失败处理和测试成本，当前阶段调试效率较低。
- 建议：先完成本地可高频测试的闭环（状态机、支付、超时、日志），房源上链放到最终阶段统一接入。
- 价值：降低迭代阻力，先保证业务正确性，再补链上增强能力。

### 11. 租房计时与自动失效（P0）
- 现状：存在 `expires_at` 字段，但缺少自动倒计时任务与自动状态迁移。
- 建议：实现定时任务，按“天/时/分”计算剩余时间；超时未支付自动取消，履约到期自动失效；房源自动转下架，商家检查后可再上架并允许调价。
- 价值：保证合同生命周期与房源状态一致，减少人工维护成本。

### 12. 合同 PDF 生成（P1）
- 现状：合同内容以 JSON 存储展示，缺少标准化可归档文档。
- 建议：在签署关键节点生成 PDF，固化合同文本、签署时间、哈希与交易信息。
- 价值：便于归档、打印、外部沟通与证据提交。

### 13. 评价模块（P2）
- 现状：无租后互评能力。
- 建议：基于已结束合同开放双向评价，支持评分与文字评价。
- 价值：沉淀信用数据，辅助后续决策。

### 14. 仲裁模块（P2）
- 现状：缺少争议受理、证据提交与处理流程。
- 建议：建立争议工单、证据上传、处理记录与结论回写机制。
- 价值：支撑纠纷闭环处理，完善平台治理。

### 15. 聊天协商模块（P2）
- 现状：房东与租客缺少站内协商渠道。
- 建议：增加签约前/履约期沟通能力，支持价格与条款协商留痕。
- 价值：降低沟通成本并保留协商证据。

### 16. 合同编辑与商家改价流程（P1）
- 现状：缺少结构化“提案-修改-确认”机制。
- 建议：支持商家发起条款/价格修改，租客确认后生成新版本并重新签署。
- 价值：避免直接覆盖原合同，保证版本可追溯。

### 17. 付款模块（P0）
- 现状：尚无完整支付闭环。
- 建议：实现押金支付、押金扣款、续租支付及交易记录与合同绑定。
- 价值：打通“签署 -> 生效 -> 履约”的核心链路。

### 22. 预付款能力（P1）
- 现状：当前版本已移除预付款实现，支付模式为一次性支付。
- 建议：后续如需恢复，需重新设计“预付款金额计算、状态流转、退款与对账”并补齐前后端接口与验真逻辑。
- 价值：可支持更灵活的履约方式，但需更强的一致性控制与风控。

### 23. 续费/续租支付能力（P1）
- 现状：当前版本已移除 `renewal` 等续费链路，仅保留 `initial` 支付。
- 建议：后续按“续租提案 -> 双方确认 -> 新周期支付 -> 状态回写”独立实现，避免与首笔一次性支付混用。
- 价值：支撑长期租赁场景，提升合同生命周期管理能力。

### 24. 租客恶意签署不付款治理（P0）
- 现状：租客可完成签署但不支付，导致合同长时间占位，影响房东成交效率。
- 建议：
  - 缩短签后待支付时限并严格自动取消（已存在基础定时任务，可增强为可配置）。
  - 增加租客“签后未付款”计数与冷却机制（例如 24h 内达到阈值后限制继续申请）。
  - 在风控视图中展示租客履约率（仅平台内部，避免直接公开隐私）。
- 价值：降低“签而不付”对房东与平台成交链路的干扰。

### 25. 反复申请签约占用房源治理（P0）
- 现状：恶意用户可反复发起申请，触发房源 `locked`，造成“可租房源被反复占位”。
- 建议：
  - 增加“同租客同房源短时重复申请”拦截与限频（如 10 分钟内不可重复申请）。
  - 增加租客维度全局申请频率控制（按小时/天限制创建合同次数）。
  - 增加异常占位自动释放策略：达到超时或风控规则立即回滚为 `available`。
  - 记录风控日志（用户ID、房源ID、IP、请求ID、触发规则）以便审计和封禁。
- 价值：防止恶意占坑，保障房源正常上架与真实租客访问公平性。

### 26. 租客与房东共同编辑合同内容（P0）
- 现状：当前合同内容主要由系统生成，缺少“双方协同补充条款”的结构化机制。
- 建议：
  - 流程改造为：`租客发起提案 -> 房东补充/修改 -> 双方确认定稿 -> 签署与上链`。
  - 增加合同草案版本模型：`draft_id / version / parent_version`，每次修改生成新版本，不覆盖历史版本。
  - 增加变更留痕：记录“修改人、修改时间、修改摘要、修改前后差异”。
  - 定稿锁定规则：进入签署阶段后，合同正文不可再编辑；如需调整必须创建新版本并重新签署。
  - 上链约束：上链 `contractHash` 必须对应“双方确认的最终定稿版本”。
- 价值：提升协商真实性与可追溯性，减少“口头约定未入合同”的纠纷风险。

### 18. 提前解约模块（P1）
- 现状：缺少提前解约申请、协商、结算与状态流转。
- 建议：支持双方发起/同意提前解约，并固化结算结果与证据。
- 价值：覆盖高频真实场景，降低线下处理成本。

### 19. 房源管理增强：修改价格（P1）
- 现状：已有创建/上下架，但价格修改未形成完整约束流程。
- 建议：增加改价接口、权限校验、操作日志，并与签约中状态联动限制。
- 价值：提升商家运营能力并避免状态冲突。

### 20. 支付后端授权签名（P0）
- 现状：当前租客可直连合约支付，若用户绕过前端直接调用合约，仍可能产生“业务状态不允许但链上发生支付事件”的孤立交易。
- 建议：引入“后端授权签名 + 合约验签”机制。支付前由后端按业务状态签发一次性授权（包含 `contractId/payer/amount/deadline/nonce`），合约仅在验签通过时接受支付。
- 价值：
  - 将“业务状态校验”前置到链上准入，阻断已取消/已过期合同被继续支付。
  - 防重放、防伪造、防跨合同冒用，提升资金与状态一致性。
  - 保留去中心化交易（用户仍自己发链上交易），同时增加风控边界。

### 27. 逾期不可支付的合约原生校验（P0）
- 现状：当前逾期拦截主要依赖后端状态机与回写接口约束，仍存在“绕过前端直接链上调用”的风险窗口。
- 建议（优先方案）：
  - 在合约中引入支付时限字段（如 `payDeadline`）与合同状态字段（如 `PendingPayment/Active/Ended/Cancelled`）。
  - 在支付函数中强制校验：`status == PendingPayment` 且 `block.timestamp <= payDeadline`。
  - 增加公开可调用的过期结算函数（如 `expireContract`），到期后将状态切为 `Cancelled/Ended` 并拒绝后续支付。
- 价值：
  - 逾期规则直接由链上执行，不再强依赖平台后端在线拦截。
  - 明确“合同文本规则”与“技术执行规则”一致，减少争议空间。
  - 为后续是否引入“平台授权签名”保留可选扩展，而非唯一方案。

### 21. MetaMask 双签名流程说明固化（P1）
- 现状：用户容易误解“签约一次签名即可完成全部流程”。
- 建议：在产品与技术文档中固定说明两步流程：
  1) 合同签约签名（意愿确认）；
  2) 支付交易签名（资金履约）。
- 价值：降低使用误解，减少“已签约但未支付/未生效”的沟通成本与误操作。

## 不在当前迭代范围
- 支付结算、押金托管、扣款争议处理。
- 多链兼容与链路抽象层重构。
- 第三方实名/CA 实际接入。
