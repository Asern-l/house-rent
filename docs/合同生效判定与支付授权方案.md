# 合同生效判定与支付授权方案

## 文档目的
- 统一“合同是否生效”的判定口径。
- 说明为何需要“后端授权签名 + 合约验签”。
- 作为后续实现与评审依据。

## 一、合同生效判定（目标口径）
合同满足以下条件时，判定为“已生效”：
1. 合同状态处于可支付阶段（`pending_payment`）。
2. 存在与该 `contractId` 绑定的首笔支付记录（`pay_type=initial` 且 `status=confirmed`）。
3. 支付方为合同租客地址。
4. 收款方为合同房东地址。
5. 支付金额满足合同首笔支付要求。
6. 支付时间在“房东签署时间 + 2 小时”窗口内。

说明：任一条件不满足，合同不应判定为生效。

## 二、当前风险点
- 当前已实现“租客直连合约支付 + 合约转付房东”。
- 但若用户绕过前端直接调用合约，可能出现“业务状态不允许、链上却发生支付事件”的孤立交易。
- 结果是：链上资金事实与链下业务状态可能短暂不一致，增加审计复杂度。

## 三、已实现方案：后端授权签名 + 合约验签

### 3.1 设计思路
1. 支付前，后端按业务规则生成一次性授权数据并签名。
2. 前端发起链上支付时，连同授权签名提交给合约。
3. 合约验签通过后，才接受支付并执行转付。

### 3.2 授权字段
- `contractId`
- `payer`（租客地址）
- `amount`
- `deadline`
- `nonce`（一次性随机数）
- `chainId`
- `contractAddress`

### 3.3 合约侧校验要点
1. 授权签名有效（`ecrecover` 或 EIP-712）。
2. 授权未过期（`block.timestamp <= deadline`）。
3. `nonce` 未被使用（防重放）。
4. 参数与交易实际值一致（`msg.sender`、`msg.value`、`contractId` 等）。
5. 合同当前状态允许支付。

### 3.4 后端侧校验要点
1. 仅对 `pending_payment` 且未超时合同签发授权。
2. 限定签发频率与有效期（短时效授权）。
3. 保留签发日志（请求 ID、签发人、时间、nonce、合同 ID）。

## 四、设计理由
1. 防绕过：阻断已取消、已过期合同的链上支付。
2. 防重放：同一授权不能重复使用。
3. 防冒用：防止跨合同、跨地址复用参数。
4. 保持去中心化：交易仍由用户钱包发起并支付 gas，平台只做准入授权。

## 五、当前实现范围
1. 已落地签署声明钱包验签与结构化证据留存。
2. 已落地支付前后端授权签名与合约验签准入。
3. 已落地合约侧支付截止时间、首笔金额、支付状态与 nonce 防重放校验。
4. 待继续完善链上事件与链下数据库的后台对账、告警和自动补偿。
